# The 100-Day Plan: From Zero → Production Python/FastAPI Backend + Agentic AI Engineer

> **Who this is for:** someone with essentially **zero** backend and AI knowledge — no OS, no networking, no HTTP, no databases, no LLM internals. Nothing is assumed. Every phase builds strictly on the previous one.
>
> **The thesis (memorize it):** *an agent is a distributed backend system with one non-deterministic component (the model).* Every backend concept you learn here directly enables an agentic one. By Day 100 you will have built, broken, secured, observed, and deployed a full production agentic platform — and you'll understand every layer of it down to packets, pages, and attention heads.

**Time budget:** ~3–4 focused hours/day for 100 days (~350 hours). Suggested split per day: **90 min theory** (generate + verify notes), **90 min build**, **30 min recall/teach-back**. If a day overflows, let it — the day numbers are a sequence, not a deadline.

**Landscape caveat (mid-2026):** framework and product names drift monthly. Concepts here are stable; re-verify specific tool versions/pricing before committing architecture to them.

**Cloud-agnostic:** this plan teaches cloud *concepts* (compute, storage, networking, identity, IaC), not one vendor. Examples name AWS/Azure/GCP interchangeably; pick whichever you have access to and map the concept onto it. No phase requires a specific cloud.

# Environment setup (do this once, before Day 1)

You're on **Windows**, but almost everything in backend/systems work assumes a **Unix** shell (`kill`, `ulimit`, `strace`, `tcpdump`, `openssl`, `dig`, `traceroute`). Fighting that on native Windows will cost you hours every phase. So:

- **Install WSL2 (Windows Subsystem for Linux) with Ubuntu, and do all builds inside it.** This gives you a real Linux environment on your Windows machine — the same OS your code will run on in production. `wsl --install` in an admin PowerShell, then reboot. This is the single highest-leverage setup step in the whole plan.
- **Every build in this plan gives Unix commands.** Run them in WSL2/Ubuntu as-written. Where a tool is genuinely Windows-native (Process Monitor, Wireshark), the plan says so.
- **Windows/PowerShell fallbacks** (only if you can't use WSL2): `kill -9 <pid>` → `Stop-Process -Id <pid> -Force`; `strace` → **Process Monitor (ProcMon)**; `tcpdump` → **Wireshark**; `ulimit` → not directly available (use WSL2 for the FD-limit exercises); `openssl`/`dig`/`traceroute` → install via `winget`/`choco` or use WSL2. Prefer WSL2 — the fallbacks are lossy.
- **Editor:** VS Code with the WSL extension (edit files on Windows, run them in Linux — best of both).

# How to run every single day (the method IS the plan)

Reading builds *recognition*; engineers at Anthropic/Google/Microsoft run on *recall + production judgment*. Run each day through these five moves:

1. **Generate the day's notes** using the prompt in `context.txt`, saved as `notes/dayNN-<slug>.md` (e.g. `notes/day10-ip-routing-ttl.md`). Tag each concept **[CORE] / [WORKING] / [AWARE]** and study to the tier. The notes must include the day's **case studies** and **2 system design concepts** (context.txt enforces this).
2. **Verify against primary sources.** For every non-obvious claim, check the official docs / RFC / PEP / paper. LLM notes contain confident subtle errors — catching them is itself training.
   - **Beginner caveat (Prereqs + Phases 0–2):** early on you can't yet catch a confident-but-wrong explanation — that's what being a beginner *is*. So for the foundational material, don't verify by open-ended searching (which surfaces more LLM-quality blog noise); verify against a **short list of trusted anchors** and trust those over the generated notes when they disagree:
     - *Python:* the official Python tutorial & docs (docs.python.org); *Fluent Python* (Ramalho) for internals.
     - *OS/computer:* *Computer Systems: A Programmer's Perspective*; Julia Evans' zines (wizardzines.com) for OS/network intuition.
     - *Networking:* Beej's Guide to Network Programming; *High Performance Browser Networking* (Grigorik, free online); the actual RFCs for HTTP (9110/9111).
     - *Concurrency/async:* the CPython docs for `asyncio`, `threading`, and the GIL; David Beazley's talks.
     - *Databases/distributed (Phase 4+):* *Designing Data-Intensive Applications* (Kleppmann) — the one book to buy.
   - From Phase 3 onward, once you have footing, widen back to open-ended primary-source verification as originally described.
3. **Do the day's Build.** No exceptions. Your git history over 100 days is your proof of skill. A concept you haven't built with, you don't know.
4. **Do the day's System Design reps.** Two per day, on paper/whiteboard, 15–20 min each: requirements → estimate → API → data → scale → failure modes. This is where topics become judgment.
5. **Recall + teach back.** Answer the notes' active-recall questions from memory; explain the day's core idea aloud in 60 seconds. Add flashcards to spaced repetition. Once per phase, **read real source code** of a tool you used (uvicorn, Pydantic, Redis, LangGraph, the Anthropic SDK).

**Rule of thumb:** if a day produced only notes — no running code, no design rep, no teach-back — the day didn't land. Redo before moving on.

# PHASE P — Prerequisites: Python, Git & the Shell (Days P1–P4)

The rest of the plan assumes you can write Python, use git, and work in a Unix shell. If you already can, skim these and skip ahead — do the P4 checkpoint to be sure. If you can't yet, **do not skip this**: every later "Build" depends on it, and struggling with syntax while also learning TCP is two problems at once. These four days are labeled `P` so they don't disturb the Day 1–100 numbering the rest of the plan cross-references.

*(These days are pure fundamentals with a low ceiling — no `### System design` or `### Case studies` blocks; those begin at Day 1. Verify against the trusted anchors in step 2 above.)*

### Day P1 — Python fundamentals I: values, control flow, functions
- **Learn:** what a variable/expression/statement is; the core types (`int`, `float`, `str`, `bool`, `None`) and that everything is an object; operators and truthiness; **control flow** (`if`/`elif`/`else`, `for`, `while`, `break`/`continue`); **functions** (`def`, arguments, defaults, keyword args, `return`, scope); string formatting (f-strings); reading tracebacks (they read bottom-up — the last line is the actual error). Just enough to write and run real programs.
- **Build:** a command-line program that solves FizzBuzz, then a small one that reads a text file and prints word counts. Run it (`python prog.py`), deliberately cause an error (call a missing variable), and read the traceback.

### Day P2 — Python fundamentals II: data structures, classes, modules
- **Learn:** the four workhorse collections and *when to use each* — `list` (ordered, mutable), `dict` (key→value, the most important one), `set` (uniqueness/membership), `tuple` (fixed record); comprehensions (`[x for x in …]`); **classes** (`class`, `__init__`, `self`, methods, attributes) and *why* they exist (bundle state + behavior); **modules & imports** (`import`, `from … import …`, what a module and a package are); exceptions (`try`/`except`/`raise`/`finally`); the standard library exists — don't reinvent it (`pathlib`, `json`, `collections`).
- **Build:** model a tiny domain in classes — a `BankAccount` with `deposit`/`withdraw` that raises on overdraft — split across two modules, with a `main.py` that imports and uses it. Write to and read back a `.json` file.

### Day P3 — Modern Python: types, virtual environments & the async syntax you'll need
- **Learn:** **type hints** (`def f(x: int) -> str:`) and why they matter (readable contracts + tooling catches bugs — the whole plan uses them; Pydantic/FastAPI are built on them); **virtual environments & packages** (why you never install globally: per-project isolation; `pip`, then meet `uv` properly on Day 7); the **`async`/`await` *syntax*** at face value (`async def`, `await something()`) — you'll learn what it actually *does* on Days 19–21, but you need to recognize it before then; f-strings, `enumerate`, `zip`, `with` (context managers) as everyday tools.
- **Build:** set up a virtual environment, `pip install requests`, and write a fully type-hinted script that fetches a public API (e.g. a joke or weather API) and prints a field from the JSON. Run `mypy` on it (install it) and fix any complaint — your first taste of the Day 7 toolchain.

### Day P4 — Git & the shell (and the P-phase checkpoint)
- **Learn:** the **Unix shell** basics you'll use daily (`cd`, `ls`, `pwd`, `cat`, `mkdir`, `rm`, pipes `|`, `>` redirection, `man`) — in **WSL2/Ubuntu** (see Environment setup); **git** from zero — what version control *is* and why (a time machine + collaboration + your 100-day proof-of-work); the core loop (`git init`, `git status`, `git add`, `git commit -m`, `git log`, `git diff`), branches (`git branch`, `git checkout -b`, `git merge`), and remotes (`git clone`, `git remote`, `git push`, `git pull`) with a GitHub account; the `.gitignore` file; *commit early, commit often* — every day's Build is a commit.
- **Build:** create a GitHub repo for this whole journey, clone it into WSL2, make your P1–P3 builds the first commits, push them. Create a branch, make a change, merge it back. **P-phase checkpoint (prove you're ready for Day 1):** from a blank file, write a type-hinted Python program with a class and a function that reads a JSON file, handles a missing-file error gracefully, and prints a result — then commit and push it *without looking anything up*. If you can do that, you're ready. If not, repeat the weak day.

# PHASE 0 — How a Computer Actually Works (Days 1–8)

Everything you'll ever debug — a slow query, a blocked event loop, an OOM-killed container, an expensive LLM context — bottoms out in CPU, memory, disk, and the OS. Learn the physics first so nothing later is magic.

### Day 1 — What a computer is: CPU, memory, and the latency pyramid
- **Learn:** why binary (two voltage states are cheap and noise-tolerant); bits/bytes; the CPU (registers, ALU, clock, the fetch–decode–execute cycle); the memory hierarchy — registers → L1/L2/L3 cache → RAM → SSD → network — and the **latency numbers every programmer should know** (≈1 ns cache, ≈100 ns RAM, ≈100 µs SSD, ≈1–100 ms network). What a "program" is (instructions in memory) and how it becomes a running process.
- **Why it exists / problem solved:** every performance decision you'll ever make (caching, batching, async, CDNs, KV-caches) is arbitrage across this pyramid — a 1,000,000× spread between RAM and a cross-ocean network call. Without these numbers, "fast" and "slow" are superstition.
- **Under the hood:** why sequential memory access beats random (cache lines, prefetch); why "CPU-bound" and "I/O-bound" are physically different problems.
- **Case studies:** ① Jeff Dean's "Latency Numbers" and how Google designs around them. ② *(stretch / aware-level — skim now, revisit after Phase 2)* LMAX Disruptor — a financial exchange built around CPU-cache mechanical sympathy to hit millions of ops/sec on one thread; don't expect to fully grasp it yet, just note that the pyramid can be an entire architecture.
- **System design (2):** *(using only today's concepts — the memory hierarchy — since you haven't met DNS, databases, CDNs, or Redis yet)* ① Order these by cost and justify with the pyramid: reading a value already in a CPU register; 1 MB read sequentially from RAM; 1 MB read randomly from RAM; a 1 MB file from SSD; 1 KB fetched from a server on another continent — then say which single one dominates a program that does one of each, and why. ② Reason about the core trade-off of the hierarchy: why can't you just "keep everything in the fastest tier"? Work through the three forces (capacity, cost per byte, volatility) that force data down the pyramid. *(The named layers that industrialize this — buffer pools (Day 37), Redis (Day 42), CDNs (Day 15) — you'll meet later; today, reason only about the principle.)*
- **Build:** Python script that measures and prints real latencies: dict lookup vs 1 MB file read vs HTTP call to a nearby site. Plot your own latency pyramid; commit it.

### Day 2 — The OS I: kernel, syscalls, and processes
- **Learn:** what an OS does (safely multiplexes hardware among programs); kernel space vs user space and why the split exists (protection); **syscalls** — the only door into the kernel (open/read/write/socket…); what a **process** is (address space + threads + file descriptors + credentials); fork/exec, exit codes, signals (SIGTERM vs SIGKILL), zombie/orphan processes.
- **Why:** containers, sandboxing, graceful shutdown, worker managers (gunicorn/k8s) — all are process management. An agent's "code execution tool" is safe *only* because of process isolation.
- **Under the hood:** trace `python -c "print('hi')"` with strace/ProcMon — count the dozens of syscalls behind one print; what a mode switch costs.
- **Case studies:** ① Chrome's process-per-tab architecture — paying memory for crash/security isolation. ② Docker demystified: a container is *just* a process with namespaces + cgroups, not a VM (you'll prove this Day 88).
- **System design (2):** ① Design a sandbox for running untrusted agent-generated code (process isolation, resource limits, no network — what layers?). ② Design graceful shutdown for a fleet of workers (SIGTERM → drain → deadline → SIGKILL).
- **Build:** spawn child processes with `subprocess`/`multiprocessing`; kill one mid-work; install a SIGTERM handler that finishes the current job before exiting. Observe a zombie and reap it.

### Day 3 — The OS II: threads, scheduling, and races
- **Learn:** thread = an execution stream *inside* a process sharing its memory; why sharing memory is fast **and** dangerous (race conditions); locks, deadlocks (the four conditions), atomicity; preemptive scheduling, time slices, run queues, **context switches** and their cost (~1–10 µs + cache pollution); why 10,000 threads collapse (the C10K problem) — the motivation for event loops (Day 21).
- **Case studies:** ① Apache (thread/process per connection) vs Nginx (event loop) — the architectural fork that C10K forced. ② The Therac-25 radiation machine — race conditions as a literal killer; why concurrency bugs are the worst class of bugs.
- **System design (2):** ① Size a thread pool for an API server (Little's law: pool ≈ arrival rate × service time; what happens when you guess wrong). ② Design a background job scheduler with priorities and starvation prevention.
- **Build:** the classic race — two threads increment a shared counter 1M times without a lock (watch it lose updates), then with a lock. Benchmark context-switch overhead: 1,000 threads vs 1,000 coroutines doing sleeps.

### Day 4 — The OS III: memory management
- **Learn:** virtual memory (every process believes it owns all RAM); pages & page tables, TLB, page faults, swap; **stack vs heap** (what lives where, and when each is freed); how allocation works; the OOM killer. Python specifics: everything is a heap object, **reference counting** + cycle-collecting GC generations, pymalloc arenas, why Python often doesn't return freed memory to the OS.
- **Why:** container memory limits, worker sizing, memory leaks, and — later — why LLM inference is *memory*-bound (the KV cache, Day 63) all live here.
- **Under the hood:** what a page fault physically triggers; why a "memory leak" in Python is usually a lingering reference, not lost memory.
- **Case studies:** ① **Instagram disabled Python's GC** to save 10% memory via shared copy-on-write pages across forked workers — read their engineering post. ② The Kubernetes OOMKill loop: a pod sized at p50 memory usage dying nightly at p99.
- **System design (2):** ① Size memory for an API deployment: per-request footprint × concurrency + headroom; set container limits that don't lie. ② Design memory budgeting for an LLM inference server (weights + KV cache per concurrent request — preview of Day 63).
- **Build:** leak memory on purpose (growing global list) and hunt it with `tracemalloc`; run a process under a hard memory cap and watch the OOM kill; fix it with streaming/chunked processing.

### Day 5 — The OS IV: files, I/O, and the descriptors that run the world
- **Learn:** file descriptors; "everything is a file" (files, sockets, pipes all read/write through FDs); buffered vs unbuffered I/O (your `write()` usually lands in the page cache, **not** on disk); `fsync` and what durability actually costs; **blocking vs non-blocking I/O**; `select`/`epoll` — the kernel facility that lets one thread watch 10,000 sockets, which *is* the engine inside asyncio (Day 21), Nginx, Redis, and Node.
- **Case studies:** ① **Postgres fsyncgate (2018)** — the kernel could silently drop fsync errors; databases worldwide were quietly less durable than believed. ② Redis persistence design — RDB snapshots vs AOF fsync policies as an explicit durability-vs-throughput dial.
- **System design (2):** ① Design a write-ahead log: why "append + fsync before ack" is the durability primitive under every database (you'll build one Day 37). ② Design log rotation + shipping for a service writing 1 GB/hour without losing lines or filling the disk.
- **Build:** write records with and without `fsync`, `kill -9` the process mid-stream, count survivors. Then lower your FD limit (ulimit), open sockets until "too many open files", and fix by closing properly / raising the limit.

### Day 6 — Python under the hood
- **Learn:** interpreted vs compiled; CPython's pipeline (source → AST → bytecode → the ceval interpreter loop); everything-is-a-PyObject (type pointer + refcount header — why a Python int is ~28 bytes, not 8); dynamic typing's cost per operation; the `dis` module; why Python is "slow" and why that mostly doesn't matter for I/O-bound backends (the network waits dwarf interpreter overhead — Day 1's pyramid).
- **Case studies:** ① Dropbox ran years of desktop + server on Python — what they optimized and when they reached for native code. ② Instagram/Meta's Cinder JIT — what it took to make Python faster at scale rather than rewriting.
- **System design (2):** ① Language-choice framework: Python vs Go vs Rust for four services (CRUD API, ML inference gateway, log processor, trading engine) — defend each. ② Design the escape hatch for CPU-bound hot paths in a Python service (native extensions, subprocess workers, a sidecar service).
- **Build:** `dis.dis()` a small function and read every opcode aloud; profile a script with `cProfile` and find the real hotspot (it's rarely where you guess); benchmark attribute lookup vs local variable in a tight loop and explain why.

### Day 7 — The modern Python toolchain + your permanent project skeleton
*(You already have this: `notes/phase0-week1.md` — review, don't regenerate.)*
- **Learn:** `uv` (env/deps/Python versions, lockfile-first), `ruff` (lint+format), `mypy --strict`, `pytest` + `pytest-asyncio`; `pyproject.toml` (PEP 621), `src/` layout, pre-commit hooks, a CI stub. Typed code as validated boundaries, not ceremony.
- **Under the hood:** uv's resolver produces a locked dependency *graph* (vs pip's sequential installs); Ruff's single-pass Rust AST is why it's 100× faster than the tools it replaced.
- **Case studies:** ① The `left-pad`/`xz` supply-chain scares — why lockfiles and pinned, auditable deps are a security control, not pedantry. ② How large Python monorepos (Google/Meta-style) enforce format/type gates in CI so review is about design, not style.
- **System design (2):** ① Monorepo vs polyrepo for a 5-service platform — trade-offs, CI implications. ② Design a dependency-upgrade policy: lockfiles, automated PRs, security patch SLAs.
- **Build:** the skeleton repo you will reuse for 93 more days — uv-managed, ruff+mypy+pytest wired into pre-commit + CI, `src/` layout, plus a hello-world FastAPI endpoint and one passing async test.

### Day 8 — Consolidation: the machine model
- **Do:** teach-backs for Days 1–7 (60 s each, aloud); re-answer all active-recall questions from memory; redo the weakest build.
- **Synthesis build:** write "The life of `python app.py`" — a 1-page trace from shell fork/exec → interpreter startup → bytecode → syscalls → FDs, from memory, then verify.
- **System design (2):** ① Design a code-execution sandbox for an AI agent, combining Days 2–5 (process isolation + memory caps + FD/network restrictions + timeouts). ② Design a single-machine job runner (queue on disk, worker processes, crash recovery) — you'll meet its distributed twin on Day 49.

# PHASE 1 — Networking From Zero (Days 9–18)

You cannot reason about APIs, microservices, CDNs, or LLM API calls without knowing what actually moves across the wire. This phase goes from voltage to `https://` — by the end you can narrate every hop of a request.

### Day 9 — What a network is: packets, frames, and switches
- **Learn:** why **packet switching** beat circuit switching (share the wire, survive failures — ARPANET's founding constraint); **encapsulation** — payload wrapped in TCP segment wrapped in IP packet wrapped in Ethernet frame (envelope-in-envelope); NICs and **MAC addresses** (layer-2 identity); switches (forward frames within a LAN) vs routers (forward packets between networks); ARP ("who has 192.168.1.7?"); the layered model (OSI names vs the real TCP/IP stack) and *why* layering exists — each layer solves one problem and trusts its neighbors.
- **Case studies:** ① ARPANET's design goal — a network that survives node loss — and how that single constraint produced packets, routing, and TCP's design. ② A broadcast storm: one looped switch flooding a LAN to death — why spanning tree and network segmentation exist.
- **System design (2):** ① Design cloud network segmentation for a 3-tier app (public subnet: LB only; private: app; isolated: DB) — this is every AWS/Azure VPC diagram. ② Design network isolation for agent tool-execution workloads (egress-restricted subnet — ties to Day 8's sandbox).
- **Build:** capture a `ping` and a `curl` with Wireshark/tcpdump; identify by hand the Ethernet frame, IP packet, TCP segment, and HTTP payload. Label the layers on a screenshot; commit it.

### Day 10 — IP: addressing, routing, TTL, and NAT
- **Learn:** IP addresses and **CIDR/subnets** (network vs host bits, compute them by hand); routing tables and hop-by-hop forwarding (no router knows the full path); **TTL** — every packet carries a hop counter, each router decrements it, at 0 the packet dies (prevents immortal packets looping forever); how `traceroute` exploits TTL=1,2,3… to map the path; fragmentation & MTU; IPv4 exhaustion → **NAT** (your laptop has no public IP; the router rewrites addresses) → IPv6; BGP in one paragraph (how networks advertise routes to each other — the internet's actual glue).
- **Case studies:** ① **Meta, Oct 2021:** a maintenance command withdrew Facebook's BGP routes — DNS servers became unreachable, every service vanished for 6 hours, and staff badges/tools failed because *everything* shared the network. ② **Pakistan vs YouTube (2008):** a national ISP's BGP hijack accidentally blackholed YouTube globally — why trust-based routing is fragile.
- **System design (2):** ① Design multi-region failover at the network level (health checks + DNS/anycast — what actually moves the traffic?). ② Design egress for private subnets (NAT gateway, IP allowlisting for third-party APIs — why your outbound calls appear from one IP).
- **Build:** `traceroute` to three sites on different continents; read every hop and its latency; watch TTL do its job. Plan a VPC on paper: carve 10.0.0.0/16 into public/private/DB subnets across 2 AZs.

### Day 11 — TCP & UDP: making packets trustworthy
- **Learn:** IP only *tries* to deliver — packets drop, duplicate, reorder. **TCP** builds reliability on top: ports (which app on this machine), the **3-way handshake** (SYN → SYN/ACK → ACK — agreeing on sequence numbers), sequence numbers + ACKs + retransmission, **flow control** (receiver window: don't drown me) vs **congestion control** (slow start/AIMD: don't drown the network), connection teardown and TIME_WAIT (why "address already in use"), **head-of-line blocking** (one lost segment stalls everything behind it). **UDP:** no guarantees, no handshake — and why that's *right* for DNS, video, and QUIC.
- **Under the hood:** what a socket really is (an FD — Day 5 — plus kernel buffers and state machine); where `send()` blocks when the window is full.
- **Case studies:** ① Why **HTTP/3 abandoned TCP for QUIC/UDP** — TCP's head-of-line blocking undermined HTTP/2 multiplexing; Google measured it at scale and rebuilt reliability in userspace. ② The classic AWS ELB idle-timeout bug: LB silently kills idle TCP connections at 60 s, app's pool doesn't notice, first query after lunch fails — a very common "random connection reset" ticket, and a good example of two correct components disagreeing about connection lifetime.
- **System design (2):** ① Design connection management between services (keep-alive pools, pool size, idle reaping, TCP vs app-level health checks — "port open" ≠ "app healthy"). ② Design a file-upload service for flaky mobile networks (resumable chunks, checksums — what TCP gives you and what it can't).
- **Build:** raw TCP client + server with Python `socket`; watch the handshake and teardown in tcpdump; `kill -9` the server mid-transfer and observe retransmits then reset. Then send one datagram over UDP and note everything that *didn't* happen.

### Day 12 — DNS: the internet's phonebook (and its most common outage)
- **Learn:** the resolution chain — stub resolver → recursive resolver (ISP/8.8.8.8) → root servers → TLD (.com) → authoritative server; record types (A/AAAA, CNAME, MX, TXT, NS, SRV); **DNS TTL** (how long each layer may cache an answer — completely different thing from IP TTL, know both cold); negative caching; DNS as a load-balancing/failover tool (round-robin, GeoDNS, low-TTL cutover); DoH/DoT in one paragraph.
- **Why:** DNS is the first hop of *every* request your backend or agent makes — and misconfigured TTLs are why "we changed the DNS but half the users still hit the old server."
- **Case studies:** ① **Dyn DDoS (2016):** the Mirai botnet took down a DNS provider — Twitter, Spotify, GitHub all "down" while their servers ran fine. Single-provider DNS = single point of failure. ② **Meta 2021 redux:** the BGP withdrawal (Day 10) mattered *because* it made Facebook's authoritative DNS unreachable — study how the two layers compounded.
- **System design (2):** ① Design a zero-downtime migration to a new server/provider using TTL staging (lower TTL days ahead → cut over → raise back). ② Design service discovery for internal microservices — DNS-based vs a registry (Consul/k8s DNS): staleness, health, trade-offs.
- **Build:** `dig +trace` a domain and annotate every step of the chain; query A/CNAME/MX/TXT records; measure cached vs uncached resolution time; override a hosts file and observe precedence.

### Day 13 — HTTP I: the protocol your whole career runs on
*(Partially covered: `notes/phase1-week3.md` — augment with the from-zero framing.)*
- **Learn:** request anatomy (method, path, version / headers / body) and response anatomy (status line / headers / body) — readable text on a TCP socket; **methods & semantics**: GET (safe), PUT/DELETE (idempotent — retry-safe), POST (neither — the dangerous one); the **status-code taxonomy** properly: 400 vs 401 vs 403 vs 404 vs 409 vs 422 vs 429; 500 vs 502 vs 503 vs 504 (each 5xx points at a *different* component); headers that matter (Host, Content-Type, Authorization, Accept); cookies and **statelessness** — HTTP remembers nothing between requests; state is *your* job (sessions, tokens — Day 32; agent memory — Day 43).
- **Case studies:** ① **Stripe's API** — status codes, error bodies, and idempotency treated as *product design*; read their API reference as literature. ② A redirect-caching disaster: shipping a `301` (permanent, cached forever by browsers) instead of `302` and being unable to undo it.
- **System design (2):** ① Design the error contract for a public API (which codes, what body shape, what's retryable — preview of RFC 9457, Day 30). ② Design URL structure + method semantics for a resource with actions that don't fit CRUD (e.g., "cancel an order" — POST to a sub-resource vs status PATCH).
- **Build:** speak raw HTTP/1.1 over your Day 11 socket by hand (type `GET / HTTP/1.1`, `Host:`, blank line); write a minimal parser that extracts method/path/headers from bytes. Feel that HTTP is just structured text on TCP.

### Day 14 — HTTPS/TLS + how HTTP evolved (1.1 → 2 → 3)
- **Learn:** the eavesdropping/tampering/impersonation problem; symmetric vs asymmetric crypto (just enough — key exchange establishes a shared secret, then fast symmetric encryption); **certificates & CAs** — the chain of trust that answers "how do I know this is really my bank?"; the TLS handshake step-by-step and what it costs (round trips — latency pyramid!); SNI. Then the evolution: HTTP/1.1 (keep-alive, but one request at a time per connection) → **HTTP/2** (multiplexed streams over one TCP connection, header compression) → **HTTP/3** (QUIC over UDP — Day 11's payoff).
- **Case studies:** ① **Heartbleed (2014)** — one bounds-check bug in OpenSSL exposed private keys across the internet; why "everyone depends on three unpaid maintainers" became a policy issue. ② **Microsoft Teams went down (Feb 2020) because a TLS certificate expired** — cert *operations*, not cert theory, is what pages you at 3 a.m.
- **System design (2):** ① Design certificate lifecycle management for 200 services (ACME/Let's Encrypt automation, rotation, expiry alerting — never manual). ② Decide TLS termination placement: at the edge/LB (simple, but plaintext inside) vs end-to-end mTLS (zero-trust, operationally heavy) — when each is right.
- **Build:** `openssl s_client -connect yoursite:443` and read the real cert chain; generate a self-signed cert and serve your FastAPI skeleton over TLS locally; watch the browser warning and explain exactly why it appears.

### Day 15 — Web caching & CDNs: not recomputing, not re-fetching
- **Learn:** caching = storing a copy closer/cheaper (Day 1's pyramid applied to the network); **`Cache-Control`** anatomy (`max-age`, `no-cache` = revalidate ≠ `no-store` = never keep, `private` vs `public`, `stale-while-revalidate`); **ETag + conditional requests** — server hashes the body, client sends `If-None-Match`, server replies `304 Not Modified` with no body; `Vary`; **CDN mechanics** — edge PoPs near users, cache keys, origin shielding, invalidation (purge vs versioned URLs), and the **cache stampede** problem (a hot key expires → thousands of requests hit origin at once).
- **Case studies:** ① **Fastly, June 2021:** one customer's config triggered a latent bug — Amazon, Reddit, gov.uk all 503 for an hour. When the cache layer *is* the front door, its failure is everyone's failure. ② **Netflix Open Connect** — Netflix puts its own cache boxes *inside ISPs*; ~95% of its traffic never touches the public internet. The logical endpoint of "move bytes closer."
- **System design (2):** ① Design global static-asset delivery (versioned/hashed filenames + immutable long TTLs + CDN — why cache invalidation becomes a non-problem). ② Design caching for a news site homepage: seconds-fresh content, million RPS — TTL vs purge-on-publish vs stale-while-revalidate, and stampede protection.
- **Build:** add real `ETag`/`If-None-Match` handling to a FastAPI endpoint; prove the `304` with curl and measure bytes saved; then simulate a stampede (cache expiry + 100 concurrent hitters) and fix it with a single-flight lock.

### Day 16 — Load balancing, proxies & the canonical trace
- **Learn:** forward proxy (speaks for clients) vs **reverse proxy** (speaks for servers — Nginx in front of your app); L4 (TCP-level) vs L7 (HTTP-level, can route by path/header) load balancing; algorithms — round robin, least-connections, IP-hash, weighted (consistent hashing preview → Day 48); **health checks** (active vs passive) and ejection; sticky sessions and why stateless + externalized state beats them (a thesis you'll meet ten more times); connection draining for zero-downtime deploys. Then assemble the whole phase: **"what happens when you type a URL"** — DNS → TCP → TLS → HTTP → CDN → LB → app → response → render, with every layer you now own.
- **Case studies:** ① **Slack, Jan 4 2021:** back-from-holidays traffic + cloud network degradation + clients auto-reconnecting in a herd → cascading overload. Load balancing, health checks, and retry storms interacting badly — read the postmortem. ② GitHub's GLB — how they do L4/L7 splitting and drain connections for deploys at scale.
- **System design (2):** ① Design the full LB stack for 1M req/s (DNS/anycast → L4 → L7 tiers → services; where TLS terminates; where health checks live). ② Design zero-downtime deployment behind an LB (draining, rolling replacement, instant rollback).
- **Build:** run 3 instances of your FastAPI skeleton behind Nginx/Caddy; load-test; `kill -9` one instance mid-test and watch health checks route around it with (near-)zero failed requests. Then write the URL-to-pixels trace **from memory** — a perennial systems-design interview question and a strong test of whether Phase 1 landed.

### Day 17 — Build an HTTP server from raw sockets (own the whole stack)
- **Learn (by building):** the accept loop; parsing requests off the socket; routing; serving responses; **concurrency models in order** — iterative (one client at a time) → thread-per-connection (Day 3) → event-loop/epoll (Day 5) — the same ladder the industry climbed; keep-alive handling; timeouts and slowloris-style abuse (a client sending one byte per minute holds your worker hostage).
- **Why:** this is *exactly* what uvicorn/gunicorn do. After today, "worker", "event loop", and "ASGI server" (Day 28) are things you've implemented, not vocabulary.
- **Case studies:** ① The C10K paper — the document that reframed server design around event loops. ② Gunicorn's worker zoo (sync/threads/async) as three answers to today's exercise — you now know which to pick and why.
- **System design (2):** ① Match concurrency model to workload for four services (static files, long-poll chat, CPU-heavy image resize, standard CRUD). ② Design abuse protection at the server level: header/body timeouts, max request size, connection caps.
- **Build:** an HTTP server on raw sockets that serves JSON and a static file, upgraded from iterative → threaded, with request timeouts. Load-test it against Day 16's Nginx setup; write down where yours dies first and why.

### Day 18 — Consolidation: networking

# PHASE 2 — Concurrency, Async Python & the Raw Agent (Days 19–26)

Backends spend their lives *waiting* (Day 1: network waits dwarf compute). This phase is how Python waits efficiently — and then you'll use it to build your first LLM agent with bare hands, no framework.

### Day 19 — Sync vs async, blocking vs non-blocking, the whole map
*(Partially covered: `notes/phase1-week2.md`.)*
- **Learn:** precise definitions — synchronous (wait here) vs asynchronous (do something else meanwhile); blocking vs non-blocking calls; **concurrency** (juggling many tasks) vs **parallelism** (many hands); I/O-bound vs CPU-bound (which resource is the bottleneck — Day 1); the three tools and their costs: **processes** (true parallelism, ~MB each, no shared memory) vs **threads** (shared memory, ~8 KB–1 MB stacks, GIL-bound in Python) vs **coroutines** (~KB, thousands cheap, cooperative).
- **Case studies:** ① Nginx vs Apache revisited — you can now explain the *architecture* behind the benchmark. ② Node.js's origin story: "what if the event loop was the only option?" — and what it made easy vs impossible.
- **System design (2):** ① Pick the concurrency model for four workloads: chat server (10k idle connections), thumbnail generator, CRUD API, ML inference gateway — justify each. ② Design a mixed workload service (mostly I/O + occasional CPU spikes) without letting the CPU work starve the I/O.
- **Build:** fetch 100 URLs three ways — sequential, `ThreadPoolExecutor`, `asyncio`+`httpx` — and produce a timing table. Explain each number using today's theory.

### Day 20 — The GIL, threading, and multiprocessing
- **Learn:** the **GIL** precisely — one thread executes Python bytecode at a time; it exists to make refcounting (Day 4) safe; it's **released during I/O** (threads are great for I/O-bound) and never during pure-Python CPU work (threads are useless for CPU-bound); `multiprocessing` — separate interpreters, true parallelism, but pickling costs and no shared memory by default; free-threaded Python (3.13+ experimental — verify current status).
- **Case studies:** ① Why every Python ML-serving stack uses process pools (or native code that releases the GIL — numpy does). ② Instagram's fork-based worker model + GC tuning (Day 4) — the same story from the process angle.
- **System design (2):** ① Design a CPU-bound image-processing pipeline in Python (process pool sizing = cores, queue in front, backpressure). ② Design an inference server: one process per GPU, a dispatcher, request batching — why this shape is universal.
- **Build:** benchmark a CPU-bound function (hashing 1M strings): single thread vs 8 threads (no speedup — GIL) vs 8 processes (~8×). Then the same with an I/O-bound task (threads win fine). Two tables, one paragraph explaining both.

### Day 21 — asyncio internals: the event loop opened up
- **Learn:** the event loop = a scheduler over an epoll selector (Day 5, full circle): ready queue, `call_soon`, timers; a **coroutine** is a function that can pause (`await`) and hand control back; **Tasks** wrap coroutines for concurrent execution; cooperative scheduling's contract — *nothing yields unless it awaits* — and its cardinal sin: a **blocking call** (`time.sleep`, sync DB driver, `requests`) inside `async def` freezes *every* task on the loop.
- **Under the hood:** what `await` compiles to (generator-based suspension); how the loop knows a socket is ready (the selector).
- **Case studies:** ① The universal postmortem shape: "one endpoint added a sync HTTP call inside an async handler; p99 went to 30 s for *all* endpoints" — learn to spot it before prod does. ② FastAPI's own docs on `def` vs `async def` handlers — the threadpool fallback exists precisely because of this trap.
- **System design (2):** ① Design an async-safe data layer (async drivers, pool limits, per-call timeouts, what happens at pool exhaustion). ② Design backpressure for a fan-in service (bounded queues, shed-load-early — reject at the edge, don't buffer to death).
- **Build:** write a mini event loop with generators (~50 lines: ready queue + a fake "socket ready" callback) to demystify the magic. Then take a real asyncio server, plant `time.sleep(5)` in one handler, load-test, and watch *every* request stall; fix with `asyncio.to_thread`.

### Day 22 — Async patterns in anger
- **Learn:** `asyncio.gather` vs `TaskGroup` (structured concurrency — children can't outlive the parent); **semaphores** for bounded concurrency (never fire 10,000 requests at once); producer/consumer with `asyncio.Queue`; **timeouts** (`asyncio.timeout`) and **cancellation** (what happens to a task mid-await; cleanup with `finally`; shielding); graceful shutdown of a task swarm.
- **Case studies:** ① Connection-pool exhaustion with httpx/aiohttp at scale — the "works locally, dies at 500 RPS" pattern and its pool-math fix. ② How well-behaved crawlers (e.g., archive.org's) implement per-host politeness — bounded concurrency + retry budgets as citizenship.
- **System design (2):** ① Design a polite concurrent crawler (global cap, per-host cap, timeout, retry budget, dedup) — the same shape as an agent calling many tools. ② Design request hedging for tail latency (send a backup request at p95; cancel the loser) and its cost trade-off.
- **Build:** a concurrent URL fetcher with semaphore(10), per-request timeouts, clean cancellation on Ctrl-C, and a progress display. This exact skeleton becomes your parallel tool-executor on Day 25.

### Day 23 — Your first LLM call: the stateless function
- **Learn:** what an LLM is at the black-box level — a function from a token sequence to a probability distribution over the next token, sampled repeatedly; **tokens** (≈ chunks of words; you pay per token in and out); the messages array and roles (system/user/assistant); **the model has no memory** — "conversation" is an illusion your backend constructs by re-sending the whole history each turn; context window limits; **the two mental models on paper before any framework:** (1) LLM call = stateless input→output function; (2) agent = a `while` loop around it (perceive → reason → act → repeat).
- **Case studies:** ① How ChatGPT-style apps actually assemble each request (system prompt + trimmed history + new message) — the "memory" is client/backend-side, always. ② A production token-cost blowup: naively re-sending unbounded history until a single conversation costs dollars per turn — the origin of every context-management technique (Day 43).
- **System design (2):** ① Design conversation storage: who owns history, where it lives (DB — Day 38's schema), what gets sent per turn. ② Design a token-budget guard: per-request and per-user caps, enforcement point, what the user sees at the limit.
- **Build:** first calls with the Anthropic SDK (consult the `claude-api` skill; model `claude-opus-4-8`): (a) prove statelessness — ask a follow-up with no history (it fails) vs with history (it works); (b) count tokens with `count_tokens` and compute the cost of a 20-turn conversation with and without history-trimming.

### Day 24 — Hand-roll a ReAct agent (no framework — ever, until you've done this)
*(Covered: `notes/phase1-week2.md` agentic half — augment.)*
- **Learn:** the **ReAct loop** — a `while` loop: send messages → model *describes* a tool call (it executes nothing — your code does) → run the tool → append the result → repeat until a final answer or a stop condition; stop conditions as a first-class design problem: **max iterations, cost ceiling, wall-clock timeout** — an agent without them is an unbounded loop with your credit card.
- **Under the hood:** the model only ever emits text; "agency" lives entirely in your loop. Every framework (Days 73–76) is this loop with opinions.
- **Case studies:** ① Early AutoGPT (2023): agents looping forever, burning API budgets, achieving nothing — the lesson that autonomy without stop conditions is a bug, not a feature. ② Anthropic's "Building Effective Agents": most production wins are simple *workflows*, not open-ended agents — the taxonomy you'll use on Day 71.
- **System design (2):** ① Design stop conditions for three agent types (support bot, research agent, code agent) — iterations, cost, time, human-escalation triggers per type. ② Design an audit log of agent decisions (every model response, tool call, and result — you'll want it for debugging, compliance, and evals).
- **Build:** a hand-rolled single-tool ReAct agent (calculator or weather) with the Anthropic SDK: manual loop on `stop_reason == "tool_use"`, `MAX_ITERS` guard, cost tracking printed per run. No framework.

### Day 25 — Tool calling internals + structured errors
*(Covered: `notes/phase1-week3.md` agentic half — augment with parallel execution.)*
- **Learn:** how tool use actually works — you send JSON-Schema tool definitions; the model emits a structured `tool_use` block (name + args); **constrained decoding** (the sampler masks tokens that would produce invalid JSON — why the args always parse); your code executes and returns a `tool_result` keyed by `tool_use_id`; **error design for a model reader** — the tool's error message is a *prompt*: "retryable: rate limited, wait" beats a stack trace; parallel tool calls executed with `asyncio.gather` (Day 22's payoff).
- **Case studies:** ① Why providers built constrained decoding (the era of "please respond with valid JSON" + regex hope, and its 5% failure tax in production). ② A production agent spiraling on a raw stack-trace error — retrying the identical call five times — versus recovering instantly once errors were made structured and actionable.
- **System design (2):** ① Design a tool registry (schemas, versioning, per-user/per-agent permissions — which agent may call which tool?). ② Design a tool error taxonomy (retryable / not-retryable / needs-human) and how the loop reacts to each class.
- **Build:** extend the Day 24 agent: two tools, one being a real HTTP call; structured LLM-readable errors on failure; parallel execution when the model requests multiple tools at once. Chaos-test: make the HTTP tool fail and watch recovery behavior with good vs bad error messages.

### Day 26 — Consolidation: concurrency + the raw agent
- **Do:** teach-backs (GIL, event loop, ReAct loop, tool protocol — 60 s each); flashcards; redo the weakest build.
- **Synthesis build:** instrument your agent to report, per run: total wall time, time waiting on the model, time waiting on tools, tokens in/out, cost. (This is primitive tracing — Day 86 makes it real.)
- **System design (2):** ① Design a multi-user agent backend on one event loop (per-user sessions, bounded concurrent agent runs, fairness). ② Design graceful degradation when the model API is down (queue? canned responses? partial service? — tie to Day 51's patterns later).

# PHASE 3 — APIs & FastAPI, Done Properly (Days 27–36)

The API layer is where your code meets the world. This phase turns "it responds to requests" into "it's a contract other engineers (and agents) can build on."

### Day 27 — What an API is + REST resource modeling
- **Learn:** an API = a **contract** (inputs, outputs, errors, guarantees) that hides implementation; why HTTP APIs won (ubiquity, statelessness, cacheability, human-readable); **REST** as constraints, not religion — resources (nouns) + uniform verbs (Day 13's methods) + statelessness; resource modeling (collections `/orders`, items `/orders/42`, relations `/orders/42/items`, actions that don't fit); the alternatives honestly: **RPC/gRPC** (typed, fast, internal), **GraphQL** (client-shaped queries, resolver complexity), when each wins.
- **Case studies:** ① Stripe/Twilio: APIs *as the product* — resource design so clean developers evangelize it. ② GitHub shipping both REST and GraphQL — why each exists and what GraphQL cost them (rate-limit complexity, resolver N+1s — Day 40's problem in disguise).
- **System design (2):** ① Model an e-commerce API's resources completely (products, carts, orders, payments, refunds) with methods + status codes per endpoint. ② Choose REST vs gRPC vs GraphQL for three contexts (public API, internal microservices, mobile BFF) — defend each.
- **Build:** write your platform's API **spec-first** as an OpenAPI document (the API you'll actually build this phase: users, sessions, agent runs, messages). Review every resource name and verb hard. Code comes tomorrow; contract comes first.

### Day 28 — FastAPI internals: what actually happens per request
*(Covered: `notes/phase1-week5.md` backend half — augment with the uvicorn/Day-17 connection.)*
- **Learn:** **WSGI vs ASGI** (sync-callable vs async-callable server↔app contracts; ASGI is why SSE/WebSockets work); the stack: **uvicorn** (the event-loop HTTP server — literally Day 17 productionized) → **Starlette** (routing/middleware) → **FastAPI** (DI + validation) → **Pydantic v2** (Rust-core validation at the boundary); **dependency injection** as a per-request dependency graph (auth, DB session, settings — resolved per request, testable via overrides); `lifespan` (create pools once at startup); the `def` vs `async def` handler rule (sync handlers get a threadpool; async handlers *must not block* — Day 21).
- **Case studies:** ① Netflix Dispatch and the pattern of FastAPI for internal platforms — why DI + types made it the Python default. ② A real "async endpoint 40× slower than sync" bug report — traced to a blocking driver inside `async def` (you predicted this on Day 21).
- **System design (2):** ① Design app structure for a 12-factor FastAPI service (settings via env, app factory, router modules, DI-provided resources). ② Design per-request context propagation (request ID, user, tenant, DB session) through DI — no globals.
- **Build:** rebuild Day 17's server as the real thing: FastAPI + uvicorn implementing your Day 27 spec skeleton. Deliberately plant a blocking call in an async handler, load-test to watch concurrency collapse, fix it, and keep the before/after numbers.

### Day 29 — Middleware, CORS & the request lifecycle
- **Learn:** the **middleware onion** — each request passes through layers inward, response outward; **order matters** (auth before handlers; error-envelope outermost; logging where it sees everything); what belongs in middleware (cross-cutting: request ID, timing, compression, auth *context*) vs handlers (business logic); **CORS demystified** — a *browser*-enforced protection (curl ignores it entirely): same-origin policy, preflight `OPTIONS`, `Access-Control-Allow-Origin`, the `*`-plus-credentials trap; why "CORS error" usually means "your server didn't say the magic words," not "blocked attack."
- **Case studies:** ① Real bug-bounty findings from reflected-origin CORS misconfigurations (server echoing any Origin with credentials allowed = any site can read your users' data). ② The `X-Request-ID` convention — how Heroku/Stripe-style platforms stitch one request's story across services (the seed of Day 85's tracing).
- **System design (2):** ① Design the middleware stack and its exact order for an API gateway (TLS → request ID → logging → rate limit → auth → CORS → compression → app) — justify every position. ② Design multi-tenant request context (tenant resolution from subdomain/header/token; isolation guarantees downstream).
- **Build:** write middleware that assigns a request ID, times the request, and emits one structured log line per request (JSON: id, method, path, status, duration). Then build a tiny HTML page on another port and fix its CORS failure properly — watch the preflight in devtools.

### Day 30 — API contracts: validation, errors, RFC 9457
- **Learn:** **validate at the boundary** — Pydantic models as the only door in; reject early with precise errors; the **error contract**: RFC 9457 (`application/problem+json` — type/title/status/detail/instance) as a consistent machine-readable shape; 400 (malformed) vs 422 (well-formed but invalid) honestly; never leak internals (stack traces are for logs, not clients); response models that guarantee output shape.
- **Case studies:** ① Zalando's API guidelines — a real company's enforced error/naming/versioning law, worth reading in full. ② An SDK-breakage incident from an error-body shape change — why error contracts are contracts too.
- **System design (2):** ① Design one error contract for a 20-microservice platform (shared problem-types registry, correlation IDs, retryability signaling). ② Design validated file upload (size caps before buffering, content-type verification, streaming to storage — don't read 5 GB into RAM; Day 4).
- **Build:** implement RFC 9457 everywhere in your API via exception handlers — including overriding FastAPI's default 422 shape into your problem format. Write contract tests asserting the error shape for every failure class (bad JSON, validation, auth, not-found, conflict, server error).

### Day 31 — Idempotency, unit of work, pagination, versioning
*(Covered: `notes/phase1-week4.md` backend half — augment with unit-of-work depth.)*
- **Learn:** **idempotency keys** — client sends a unique key with a side-effecting POST; server stores key→response and replays it on retry (the *only* cure for "did my payment go through?" double-clicks and network retries); the **unit of work** pattern — one request = one atomic unit: all its writes commit together or none do (the transaction boundary — full mechanics Day 39); **pagination**: offset (skips/duplicates under concurrent writes, slow at depth) vs **cursor** (opaque token = stable position); **versioning** (URL vs header; additive-change discipline so you rarely need v2).
- **Case studies:** ① **Stripe's idempotency keys** — the canonical design: key storage, replay semantics, TTL; read their docs + engineering blog. ② Shopify's API versioning (dated versions, quarterly, deprecation windows) as a sustainable public-API evolution policy.
- **System design (2):** ① Design a payment-creation endpoint that survives client retries, double-clicks, and timeouts (idempotency store, in-flight concurrency, response replay). ② Design cursor pagination for a hot activity feed (cursor contents, index backing it — Day 40 — and why offset breaks here).
- **Build:** an idempotency-key middleware backed by a store (in-memory today; Redis on Day 42): same key → replayed response, concurrent same-key → one execution. Plus a cursor-paginated list endpoint. Chaos-test with a client that fires every request twice.

### Day 32 — AuthN & AuthZ: who are you, what may you do
*(Covered: `notes/phase1-week5.md` auth half — augment.)*
- **Learn:** authentication vs authorization; password storage (bcrypt/argon2 — slow *on purpose*); **sessions** (stateful, revocable) vs **tokens** (stateless, scalable, revocation is hard) — the trade-off, not a winner; **JWT internals** — header.payload.signature; *decoding is not verifying* (the signature check is the security boundary); claims (exp/aud/iss) and pitfalls (alg=none, no revocation, don't stuff PII); **OAuth 2.0 + OIDC** — the authorization-code-with-PKCE flow end to end (what each redirect and token exchange does; access vs refresh vs ID token); **RBAC vs ABAC**; **BOLA** preview — "does this user own this object?" checked on *every* access (the #1 real-world API vuln — Day 89).
- **Case studies:** ① **Facebook's 2018 access-token breach** (50M tokens minted via a "View As" bug chain) — why token lifetime and revocation strategy matter. ② The recurring `alg=none` / algorithm-confusion JWT vulnerabilities across libraries — why you pin the algorithm and verify, always.
- **System design (2):** ① Design auth for a multi-tenant B2B SaaS (SSO/OIDC per tenant, role model, token lifetimes, refresh rotation, logout semantics). ② Design service-to-service auth (mTLS vs signed tokens vs cloud-managed identities) for a 10-service platform.
- **Build:** JWT auth on your API via DI (`get_current_user`): login issues tokens, protected routes verify. Then write attack tests: expired token, tampered payload, wrong audience, `alg` swap — all must fail with your RFC 9457 errors.

### Day 33 — Rate limiting, streaming & long-running work
- **Learn:** **rate-limiting algorithms** — token bucket (allows bursts, refills steadily), leaky bucket (smooths), fixed window (boundary flaw: 2× at the edge), sliding window (fixes it); what to key on (user/API key/IP/tenant) and returning `429` + `Retry-After` honestly; **streaming**: SSE (one-way, plain HTTP — perfect for LLM tokens) vs WebSockets (bidirectional — Day 95); **the long-running-work pattern**: never hold an HTTP connection for minutes — return `202 Accepted` + job ID, expose status endpoint, or push via webhook (the queue behind it arrives Day 49).
- **Case studies:** ① GitHub's rate-limit design (headers exposing limit/remaining/reset — limits as a *communicated contract*). ② Why every LLM provider streams (SSE) — time-to-first-token is UX; total time is the same but perceived latency drops massively.
- **System design (2):** ① Design tiered, distributed rate limiting (per-user + per-tenant + global; where state lives — Redis, Day 42; fail-open or fail-closed when the limiter is down?). ② Design an async-job API (submit → 202 + ID → status polling vs webhooks; job states; result retention).
- **Build:** a sliding-window rate limiter as middleware (in-memory; you'll port it to Redis Day 42) returning proper 429s; an SSE endpoint that streams a Claude response token-by-token (`AsyncAnthropic` + `StreamingResponse`).

### Day 34 — The agent behind an API (integration day)
- **Learn (by integrating):** everything this phase built, wrapped around the Phase-2 agent: auth on agent endpoints; DI-scoped per-request agent instances (per-user history, per-user credentials); SSE streaming of agent steps; per-user rate limits and **spend guards** (tokens = money — a cost ceiling per user/day is a production requirement, not a nicety); the audit log from Day 24 persisted properly.
- **Case studies:** ① The public pattern of LLM-app cost incidents (a leaked/unlimited endpoint hammered for free inference) — why auth + rate limits + spend caps are the first three bricks. ② How coding-assistant backends shape their agent APIs (session create → streamed turns → tool events) — you're building the same shape.
- **System design (2):** ① Design a multi-tenant agent API (session isolation, per-tenant model config, per-tenant spend caps, noisy-neighbor control). ② Design degradation tiers for model-API slowness (tight timeouts → smaller model fallback → queue for later → honest 503 — tie to Day 51 later).
- **Build:** the production-shaped agent endpoint: `POST /sessions` + `POST /sessions/{id}/messages` (SSE-streamed agent turn: model tokens + tool events), JWT-protected, rate-limited, spend-capped, fully audit-logged. This service is your platform's core from now on.

### Day 35 — Reasoning strategies: why they work mechanically
*(Covered: `notes/phase1-week4.md` agentic half — augment.)*
- **Learn:** **Chain-of-Thought** — the intermediate tokens *are* the computation (the model has no scratchpad except the tokens it emits; more reasoning tokens = more forward passes before committing); **ReAct** (interleave reasoning with tool acts — you built it); **Plan-and-Execute** (plan once, execute steps — cheaper, less adaptive); **Reflexion/self-critique** (a second pass judging the first); **self-consistency** (sample N answers, majority-vote); **Tree-of-Thought** (explore branches — costly, for search-shaped problems); the economics: each strategy is a token/latency/quality trade you should choose deliberately per task.
- **Case studies:** ① The original CoT results — the same model jumping dramatically on math/logic benchmarks with "let's think step by step"-style elicitation; what that implies about latent capability. ② Production copilots (research/coding assistants) using plan-then-execute with a visible plan a human can veto — structure as a safety *and* quality tool.
- **System design (2):** ① Match strategy to task for four products (support triage, code generation, financial research, data extraction) with a cost ceiling per task. ② Design a "thinking budget" policy — when does a request earn more reasoning (adaptive thinking / effort), and who decides (router? user tier? task type?).
- **Build:** implement Plan-and-Execute alongside your ReAct agent; run both on the same 10-task suite; produce the table: success rate, tokens, latency, cost. Draw the conclusion you'd defend to a team.

# PHASE 4 — Data: Databases, Caching, Vectors & RAG (Days 37–46)

State is the hard part of every system — and *all* agent "memory" is just state with a retrieval strategy. Learn how data is actually stored, found, and kept correct; then build retrieval for meaning, not just equality.

### Day 37 — How a database actually works (before SQL)
- **Learn:** why not just files (concurrent writers corrupt; crashes tear writes; finding one record = scan everything); the storage engine: **pages** (~8 KB units), the **buffer pool** (hot pages cached in RAM — Day 1's pyramid again), the **WAL** (append + fsync the *intent* first, so a crash mid-write is recoverable — Day 5's payoff), **B-trees** (the disk-friendly sorted tree behind almost every index); the query lifecycle: parse → plan (the optimizer *estimates*) → execute.
- **Case studies:** ① SQLite — the most deployed software on earth (every phone/browser); read its "How It Works" docs; a full WAL+B-tree engine in one file. ② Postgres's WAL as a platform: replication, point-in-time recovery, and CDC (Day 52) all ride the same log.
- **System design (2):** ① Design a KV store from first principles: append-only log + in-memory index → what compaction and crash recovery require (you've invented Bitcask/the LSM seed). ② Choose durability settings per workload (`synchronous_commit` off/on/remote) for: analytics, orders, chat messages — with honest data-loss windows.
- **Build:** a toy persistent KV store in Python — append-only log file + in-memory hash index + `fsync` on write + startup recovery by log replay + a compaction routine. Crash-test it with `kill -9` mid-write.

### Day 38 — SQL & data modeling
- **Learn:** the relational model (tables/rows, keys, foreign keys); joins as the superpower; **normalization** (1NF→3NF — store each fact once so it can't disagree with itself) and **deliberate denormalization** for read paths; constraints (NOT NULL, UNIQUE, CHECK, FK) as guardrails that make bad states unrepresentable; migrations as versioned schema evolution (alembic).
- **Case studies:** ① GitHub's online schema migrations (gh-ost) — changing a billion-row table with zero downtime; why "just ALTER TABLE" stops working at scale. ② A denormalization bite: cached `order_total` disagreeing with line items — when you duplicate data, you sign up to keep it consistent.
- **System design (2):** ① Model a chat application's schema (users, conversations, messages, receipts, presence?) — decide what's relational and what isn't. ② Design multi-tenancy at the data layer: shared-schema+tenant_id vs schema-per-tenant vs DB-per-tenant (isolation vs ops cost vs noisy neighbors).
- **Build:** design and migrate your platform's real schema in Postgres: users, sessions, messages, tool_calls, audit_log — FKs, constraints, seed data, alembic migrations. Rewire Day 34's service onto it.

### Day 39 — Transactions: ACID, isolation, MVCC
- **Learn:** **ACID** precisely (Atomicity: all-or-nothing via WAL; Consistency: constraints hold; Isolation: concurrent ≈ serial; Durability: fsync'd WAL); the **anomalies** — dirty read, non-repeatable read, phantom read, **lost update** (the one that eats money); **isolation levels** (Read Committed → Repeatable Read → Serializable) as an explicit correctness-vs-throughput dial; **MVCC** (writers create new row versions; readers see a snapshot; readers never block writers); locks (row/table, `SELECT FOR UPDATE`), deadlocks and consistent lock ordering; the Day 31 **unit of work** finally mechanized: request → one transaction → commit or rollback.
- **Case studies:** ① The classic double-withdrawal: two concurrent requests read balance=100, both withdraw 100 — fixed by `FOR UPDATE` or serializable isolation, not by hope. ② ORM-induced deadlock storms (two code paths locking the same rows in opposite order) and the lock-ordering fix.
- **System design (2):** ① Design checkout for limited inventory (no overselling under 1,000 concurrent buyers: atomic decrement vs lock vs serializable — compare). ② Design optimistic concurrency for a document editor (version column, compare-and-swap, retry loop) vs pessimistic locking — when each.
- **Build:** open two `psql` sessions and *reproduce every anomaly* at the isolation level that allows it, then raise the level and watch it vanish. Cause a real deadlock, read the error, fix with lock ordering. Wrap one multi-write endpoint in a proper unit of work.

### Day 40 — Indexing & query performance
- **Learn:** what an index is (a B-tree from column values → row locations — Day 37); what it accelerates (equality, range, prefix — *not* `%contains%`); **composite index column order** (leftmost-prefix rule); covering indexes; partial indexes; **every index taxes every write**; reading **`EXPLAIN ANALYZE`** as a daily habit (seq scan vs index scan; estimated vs actual rows — when the planner guesses wrong); the **N+1 problem** (a loop issuing one query per item — one of the most common performance bugs in backend code; fix with joins/eager loading); **connection pooling** (connections are expensive; pool math: size ≈ what the DB can actually serve, not what the app wants; pgbouncer for many-worker fleets).
- **Case studies:** ① The evergreen "missing index took the site down at 10× traffic" postmortem shape — seq scans that were invisible at 1,000 rows, fatal at 10M. ② An ORM N+1 story: a dashboard issuing 1,201 queries per page load, fixed to 3 with eager loading — and how it got *detected* (query logging in dev).
- **System design (2):** ① Design the index plan for a feed query (`WHERE tenant_id = ? AND created_at > ? ORDER BY created_at DESC LIMIT 50`) — which composite index, in which order, and why it also serves your Day 31 cursor. ② Design DB access for 500 ephemeral workers (Lambda-style) against one Postgres — pooler in transaction mode, pool sizes, and what breaks without it.
- **Build:** seed a 10M-row table; write a realistically slow query; fix it with `EXPLAIN ANALYZE` + the right composite index and record the before/after (ms and plan). Reproduce an N+1 in SQLAlchemy against your schema; fix with eager loading; count queries both ways.

### Day 41 — NoSQL models & the CAP theorem
- **Learn:** the four NoSQL shapes and the query patterns they serve — document (nested aggregates), key-value (Day 37's build!), wide-column (write-heavy time-series), graph (relationship traversal); **CAP correctly**: only during a network partition must you choose — refuse writes (**C**) or accept divergence (**A**); **PACELC** (even with no partition: latency vs consistency); consistency models (strong, eventual, causal, **read-your-writes**); replication lag as the everyday "eventual" you'll actually meet; and the honest default: **Postgres until proven otherwise** (it does JSONB, full-text, and — Day 44 — vectors).
- **Case studies:** ① **Amazon's Dynamo paper** — the shopping cart that must *never* refuse a write, choosing A + conflict resolution; the paper that launched the NoSQL era. ② **Google Spanner** — choosing C with atomic-clock-bounded time (TrueTime), and what that buys (external consistency) at what cost. Read both as the two poles of the same trade.
- **System design (2):** ① Pick stores for five workloads — user sessions, financial ledger, product catalog, social graph, clickstream analytics — and defend each against "why not Postgres?". ② Design read-your-writes UX on an eventually consistent read-replica setup (session pinning, write-then-read routing, version tokens).
- **Build:** model your platform's `sessions+messages` both ways — normalized relational vs one JSONB document per session; benchmark the read and write patterns; write a paragraph on when the document wins. Then demonstrate replica lag: write to primary, read stale from your Day-47-preview replica (or simulate), and fix with pinning.

### Day 42 — Caching in anger + Redis internals
- **Learn:** the caching patterns — **cache-aside** (default: read cache → miss → read DB → fill), write-through, write-behind (fast, risky); **invalidation** honestly (the two hard things): TTL (bounded staleness) vs explicit invalidation on write (precise, complex) vs event-driven (Day 52's outbox+CDC); **cache stampede** protection (single-flight locks, stale-while-revalidate — Day 15 at the app layer); **Redis internals**: single-threaded event loop (Days 5+21 — that's *why* every command is atomic), data structures (strings/hashes/lists/sets/sorted-sets/streams), eviction policies (LRU/LFU/TTL), persistence (RDB vs AOF — Day 5's fsync dial again); Redis roles: cache, session store, rate limiter, distributed lock (and redlock's caveats), queue.
- **Case studies:** ① **Facebook's "Scaling Memcache" paper** — leases, stampede control, and cross-region invalidation at billions of ops/sec; the canonical cache-at-scale document. ② A thundering-herd postmortem: one hot key expiring → database flattened → site down; fixed with jittered TTLs + single-flight.
- **System design (2):** ① Design caching for a product page at 100k RPS (key design, TTL+jitter, invalidation on price change, stampede guard, cold-start warming). ② Design a *distributed* rate limiter on Redis (atomic Lua sliding window, fail-open vs fail-closed, clock concerns) — then implement it.
- **Build:** add Redis to the platform: cache-aside on your slowest endpoint (measure hit ratio + latency delta), port Day 33's rate limiter to Redis (atomic via Lua/INCR+EXPIRE), move idempotency keys (Day 31) to Redis with TTL. Break it: restart Redis mid-traffic and observe both fail-open and fail-closed behavior.

### Day 43 — Agent memory: state with a retrieval strategy
- **Learn:** the memory taxonomy — **working** (the context window itself — the only thing the model ever sees), **episodic** (past sessions — rows in your DB), **semantic** (knowledge — the RAG store, Days 44–45), **procedural** (instructions/skills — prompts and code); **context-window management**: sliding windows, running summarization (compress old turns), structured scratchpads (facts extracted to fields, not prose); "context rot" — long contexts degrade attention quality (mechanism on Day 62); why long chats get slow and expensive (every token reprocessed every turn — the KV-cache economics, Day 63).
- **Case studies:** ① ChatGPT's memory feature — extracted durable facts, user-visible and editable: episodic → semantic promotion with consent as *product* design. ② Claude Code's auto-compaction — summarize-and-continue when the window fills: context management as an engineering feature you'll now build.
- **System design (2):** ① Design session memory for a support agent (what persists verbatim vs summarized vs extracted; token budget per turn; PII controls). ② Design cross-session user memory (what's remembered about a user, where it's stored, how it's injected, how users delete it — privacy by design).
- **Build:** give your platform agent real memory: full history in Postgres (episodic — you have the schema); a token-budgeted context assembler (recent turns verbatim + older turns summarized by a cheap model call); measure tokens/turn before vs after on a 50-turn conversation.

### Day 44 — Embeddings & vector search
- **Learn:** the problem — SQL finds *exact/range* matches; nothing so far finds "similar meaning"; **embeddings** — a model maps text to a dense vector (~hundreds–thousands of dims) where semantic similarity ≈ geometric closeness; **cosine/dot-product similarity**; exhaustive search is O(N·d) — fine to ~100k, then you need **ANN**: **HNSW** (a multi-layer navigable graph — express lanes down to local streets; sub-linear search) and its recall-vs-latency/memory dials (`ef`, `M`); embedding model choice (dimensions, domain, multilingual) and the fact that **vectors are model-versioned** (change embedder = re-embed everything); `pgvector` — vectors living next to your relational data (no new infra until proven necessary — Day 41's honesty).
- **Case studies:** ① Spotify's Annoy — early production ANN for music recommendations; the lineage from there to HNSW-everywhere. ② Semantic search at Glean/Notion-scale — why keyword search fails on paraphrase ("PTO policy" vs "vacation days") and what embedding search recovered.
- **System design (2):** ① Design semantic search over 10M documents (index choice, HNSW params, metadata pre-filtering vs post-filtering — the filtered-ANN trap). ② Design an embedding versioning/migration strategy (model upgrade → dual-write + background re-embed + cutover, without downtime).
- **Build:** `pgvector` on your Postgres: embed a real document set (your own notes!), HNSW index, similarity endpoint. Measure recall@10 of HNSW vs brute-force on 1k queries; turn the `ef` dial and plot recall vs latency.

### Day 45 — The RAG pipeline: chunking, hybrid search, reranking
- **Learn:** why RAG (models don't know your data; contexts are finite; fine-tuning doesn't reliably add facts — Day 66); the pipeline: **ingest → chunk → embed → index → retrieve → rerank → assemble**; **chunking** — the #1 cause of bad RAG (fixed-size cuts destroy meaning; structure-aware/semantic chunking preserves it; overlap; chunk-size trade-offs); **hybrid search** — dense vectors miss exact identifiers (error codes, SKUs, names) that sparse **BM25** nails; fuse with **RRF**; **reranking** — bi-encoder retrieval is fast-but-approximate, a **cross-encoder** reads query+doc *together* for precision on the top-50 (usually the highest-ROI upgrade); context assembly with **citations** (grounding = auditability); **RAG caching**: embedding cache, retrieval cache, and **semantic answer cache** (similar-enough question → cached answer) + invalidation when the KB changes (Day 42's discipline).
- **Case studies:** ① **Morgan Stanley × GPT-4** — RAG over ~100k wealth-management documents with citation-grounded answers for advisors; the enterprise-RAG reference story. ② A typical production RAG postmortem: "the model hallucinates" → actually retrieval returned garbage → actually *chunking* split every policy mid-sentence; fixed at ingest, not at the model.
- **System design (2):** ① Design RAG over a company wiki with daily edits (ingest pipeline, incremental re-index, cache invalidation on update, freshness SLA). ② Design citation-grounded answers for a regulated domain (chunk-level provenance, confidence thresholds, "I don't know" paths, audit log).
- **Build:** the full pipeline over a real corpus: structure-aware chunking → embeddings + BM25 (Postgres FTS) → RRF fusion → cross-encoder rerank → assembly with citations. Build a 30-question **golden Q&A set** and measure retrieval hit-rate: dense-only vs hybrid vs hybrid+rerank (three-row table — keep it, Day 80 reuses it). Add a semantic answer cache with invalidation on document update.

### Day 46 — Agentic RAG + consolidation
- **Learn:** **static RAG** (retrieve once, answer) vs **agentic RAG** (retrieval is a *tool* — the agent can reformulate queries, retrieve again if results look bad, and decide when it has enough) — better on hard questions, more tokens/latency; **GraphRAG** overview (entity/relationship extraction → graph traversal — for genuinely relational corpora); **vector-store selection as a CAP/ops decision** (pgvector vs Qdrant/Pinecone/Azure AI Search vs OpenSearch: consistency, filtering power, ops burden, cost) — the store is a dependency like any other (pooling, timeouts, Day 51 resilience).
- **Case studies:** ① Microsoft's GraphRAG results on "global" questions (themes across a whole corpus) where vanilla RAG fails structurally. ② Deep-research-style products (multi-step retrieve→read→reformulate loops) as agentic RAG at production scale.
- **System design (2):** ① Design the agentic-RAG loop with budgets (max retrievals per question, result dedup, early-exit conditions, cost ceiling). ② Design end-to-end KB freshness (doc updated → re-chunk → re-embed → caches invalidated → agent sees new fact) with a measurable SLA.
- **Build:** convert Day 45 into agentic RAG (retrieval as a tool of your platform agent, reformulation allowed, budget-capped); A/B against static RAG on the golden set: quality / tokens / latency / cost table. **Phase teach-backs:** WAL, MVCC, N+1, CAP, cache invalidation, chunking→rerank — 60 s each, aloud.

# PHASE 5 — Distributed Systems & Durable Execution (Days 47–58)

One machine can't scale forever and *will* fail. The moment you have two machines you have partial failure, ordering, and consistency problems. This phase is the heart of senior backend engineering — and the direct blueprint for multi-agent systems.

### Day 47 — Why distributed systems are hard: replication, sharding, partial failure
- **Learn:** why distribute (scale beyond one box, survive its death, be near users); the **fallacies of distributed computing** (the network is *not* reliable, latency is *not* zero…); **partial failure** — the defining nightmare: A called B, got no answer; did B do the work? (this single ambiguity motivates idempotency, retries, sagas, and durable execution); clocks lie (NTP drift; never order events by wall-clock across machines; logical clocks in one paragraph); **replication** (leader-follower, WAL shipping — Day 37; replication lag; failover and split-brain); **sharding** (hash vs range; hot keys; resharding pain — Day 48 fixes); the ur-thesis: **stateless services + externalized state** scale horizontally.
- **Case studies:** ① **GitHub, Oct 21 2018:** a 43-*second* network partition → cross-DC MySQL split-brain → 24 hours of degraded service and careful data reconciliation. Read the postmortem slowly; it's the whole phase in one incident. ② Instagram's ID sharding scheme (timestamp + shard + sequence packed in 64 bits) — practical sharding you can hold in your head.
- **System design (2):** ① Shard a 2 TB multi-tenant Postgres (shard by tenant; routing layer; the cross-shard query problem; resharding plan). ② Design read-replica routing with lag awareness (which reads may be stale; pinning after writes — Day 41's read-your-writes, now mechanized).
- **Build:** docker-compose a Postgres primary + streaming replica; measure replication lag under write load; demonstrate a stale read-after-write against the replica, then fix it with write-then-pin routing in your API's DB layer.

### Day 48 — Consistent hashing & load distribution
- **Learn:** the problem — `hash(key) mod N` remaps ~everything when N changes (cache wipeout on every scale event); **consistent hashing** — nodes and keys on a ring; a node change remaps only ~1/N of keys; **virtual nodes** (each physical node = many ring points → even spread); rendezvous hashing as the simpler alternative; where it's load-bearing: distributed caches, DB/queue partitioning (Kafka's partitioner — tomorrow), sticky routing for stateful connections (WebSockets — Day 95).
- **Case studies:** ① The original Akamai/Karger paper — consistent hashing was invented *for* CDN cache distribution (Day 15 full circle). ② **Discord** sharding millions of concurrent users' guilds across nodes via consistent hashing — read their scaling posts.
- **System design (2):** ① Design a distributed cache tier that survives node loss and scale-out with minimal hit-rate damage (ring + vnodes + replication of hot ranges). ② Design WebSocket connection routing (a user's connection is *state* — consistent-hash users to gateway nodes; what happens on node death — preview Day 95).
- **Build:** implement consistent hashing with virtual nodes in Python (~100 lines); distribute 1M keys over 10 nodes; kill a node and measure keys remapped (~10%) vs mod-N (~90%); plot distribution evenness vs vnode count.

### Day 49 — Messaging: queues, pub/sub, and the log
- **Learn:** why a queue (decouple producer from consumer; absorb bursts — Day 3's Little's law; retry without the user waiting; the answer to Day 33's `202` pattern); the three shapes — **work queue** (each message → one consumer; RabbitMQ), **pub/sub** (each event → all subscribers; fan-out), **the log** (Kafka: append-only, offsets, consumer groups, *replayable history* — event sourcing's substrate, Day 52); **RabbitMQ mechanics** (exchanges/bindings/queues, **acks** — Day 47's partial-failure answer: unacked messages redeliver, prefetch/QoS); **Kafka mechanics** (partitions = ordering + parallelism unit, keyed partitioning — Day 48!, consumer groups, retention); background jobs (Celery/arq) = ergonomics over a queue.
- **Case studies:** ① **Kafka's origin at LinkedIn** — unifying N×M point-to-point pipelines into one durable log; the paper/blog is the "why a log" argument itself. ② **Shopify flash sales** — queues absorbing 10× spikes between checkout and fulfillment: buy processing time, keep the front door responsive.
- **System design (2):** ① Design order processing (checkout API → queue → fulfillment workers): queue depth as your health metric, worker autoscaling on lag, poison-message plan (tomorrow's DLQ). ② Choose Rabbit vs Kafka vs Redis Streams for three scenarios (task offloading, event broadcast to 5 services, clickstream with replay) — defend each.
- **Build:** RabbitMQ producer + consumer with manual acks + prefetch; kill the consumer mid-message and watch redelivery. Then the same flow on a Kafka-style log (Redpanda): consumer groups, then **rewind offsets and replay history** — feel the difference between a queue and a log.

### Day 50 — Delivery guarantees & background workers
- **Learn:** **at-most-once** (ack before work: crash = lost) vs **at-least-once** (ack after work: crash = duplicate) — you *cannot* have exactly-once delivery over an unreliable network (Day 47's two-generals ambiguity); the industry answer: **exactly-once *effect* = at-least-once delivery + idempotent consumer** (Day 31's payoff — this sentence is a top interview answer); visibility timeouts; **poison messages** & **dead-letter queues** (after N failures, quarantine — never let one bad message stall the world); job-status APIs completing Day 33's `202` pattern.
- **Case studies:** ① The "duplicate email/charge sent 3×" classic — worker crashed after send but before ack; redelivery did the rest; idempotency key on the *effect* is the fix. ② Segment's exactly-once pipeline writeups — dedup keys + idempotent sinks at billions of events/day: the pattern industrialized.
- **System design (2):** ① Design payment-webhook processing with exactly-once *effect* (signature check → dedup store → idempotent handler → ack; replay-safe end to end). ② Design DLQ operations (alerting on depth, triage UI, fix-and-replay workflow, poison-classification).
- **Build:** a worker that crashes 30% of the time mid-job: observe duplicates under at-least-once; make the handler idempotent (Redis dedup key from Day 42); add max-retries → DLQ → a replay script. Wire the job-status endpooint so a client can follow their job through all of it.

### Day 51 — Resilience patterns: designing for failure
- **Learn:** **timeouts everywhere** (an unbounded wait is a resource leak — every network call gets one, end of discussion); **retries done right** — only idempotent ops, budgeted, **exponential backoff + jitter** (synchronized retries are a self-inflicted DDoS: the thundering herd — jitter decorrelates them); **circuit breakers** (after K failures, fail *fast* for a cooldown; half-open probes recovery — protects both caller and the struggling callee); **bulkheads** (partition pools per dependency so one sick dependency can't drain everything); **load shedding** (reject early at the edge beats buffering to death); **graceful degradation** (brownout: drop expensive features, keep the core). Applied verbatim to agents: every tool call and model call gets timeout + budgeted-retry + breaker; an agent retry-looping a broken tool is a thundering herd with a credit card.
- **Case studies:** ① **Netflix Hystrix** — the library that mainstreamed breakers/bulkheads; study the *concepts* doc (the successor is resilience4j, the ideas are eternal). ② **AWS's "Exponential Backoff and Jitter" post** — the actual math showing full-jitter winning; short, canonical, read it. ③ (bonus) Slack Jan 2021 (Day 16) reread — you can now name every missing pattern.
- **System design (2):** ① Write the resilience spec for *every* outbound call in your platform (DB, Redis, model API, tools, queue): timeout, retry budget, breaker thresholds, fallback per dependency — as a table. ② Design brownout modes for the agent product under model-API degradation (smaller model → cached/canned answers → queue-for-later → honest 503, with automatic tier transitions).
- **Build:** a `resilient_call` wrapper (timeout + full-jitter backoff + circuit breaker + metrics) and wrap every agent tool and the model client with it. Chaos-test: make a tool fail 100% for 60 s — watch the breaker open, the agent get a fast structured failure (Day 25's error taxonomy), and recovery probe when it heals. Global loop guard: max iterations + cost ceiling + wall-clock cap.

### Day 52 — Distributed data patterns: outbox, saga, CQRS, event sourcing
- **Learn:** the **dual-write problem** (DB commit + queue publish = two systems, no shared transaction; crash between them = lie to one of them) → the **outbox pattern** (write business change + event to *one* DB transaction; a relay/CDC ships events out — atomicity restored); **CDC** (Debezium-style: tail the WAL — Day 37 pays off again); the **saga pattern** — a distributed transaction as a sequence of local transactions, each with a **compensating action** (undo semantically, not magically: refund, release, cancel); orchestrated vs choreographed sagas; **CQRS** (separate write model from read-optimized projections); **event sourcing** (state = fold of an append-only event log — Kafka Day 49, and *exactly* what an agent's message history is); and the honest counsel: reach for these only when a plain transaction can't span the boundary.
- **Case studies:** ① Uber's trip/payment workflows — saga-shaped multi-service flows with compensations at massive scale (their engineering blog + the pattern literature). ② Debezium+outbox adoption as *the* standard answer to "how do I update the cache/search index when Postgres changes?" — which is literally your Day 42/45 invalidation problem, solved properly.
- **System design (2):** ① Design order + inventory + payment as an orchestrated saga: every step's compensating action, idempotency (Days 31/50), and the "compensation itself fails" plan. ② Design outbox+CDC to keep Redis (Day 42) and your RAG index (Day 45) consistent with Postgres — no more dual writes anywhere in your platform.
- **Build:** implement the outbox: business write + event row in one transaction; a relay process ships events to RabbitMQ; consumers update the cache. `kill -9` the relay mid-batch and prove no event is lost (at-least-once + idempotent consumer — everything converges). Then a 3-step saga (reserve → charge → fulfill) where step 2 fails and compensation unwinds step 1.

### Day 53 — Durable execution: the keystone
- **Learn:** the problem — a multi-step workflow (or agent run) crashes at step 7 of 10: restart from scratch (redo side effects?!) or resume from 7? **Durable execution** (Temporal / Azure Durable Functions): the orchestrator function's progress is captured as an **event history**; on crash/restart the code is **replayed** from the start, with completed steps returning recorded results instantly from history, until it reaches the first un-executed step — the program *appears* to resume mid-function (event sourcing — Day 52 — applied to program state); the discipline it demands: **orchestrator code must be deterministic** (no direct clock/random/network — push those into activities); timers, heartbeats, and **external events** (durable waits — a workflow paused weeks for an approval, surviving deploys); workflow versioning.
- **Case studies:** ① Temporal's lineage (Uber Cadence → Temporal) and public adopters (Snap/Netflix/Coinbase-class — verify current list) for payments, onboarding, infra orchestration. ② Azure Durable Functions' replay model — the same concept, serverless form; its docs' determinism constraints are the cleanest statement of *why* replay works.
- **System design (2):** ① Design a refund workflow that survives crashes and deploys (durable steps, compensations from Day 52, idempotent activities, human-review external event for large amounts). ② Design "wait for human approval, up to 7 days, then escalate" as a durable timer + external event race — zero processes held open.
- **Build:** the crash test you must not skip: a 5-step workflow (Temporal Python SDK, or a ~150-line replay engine of your own: event log + deterministic replay — building it teaches the most); `kill -9` the worker at step 3; restart; **prove** steps 1–2 do not re-execute and the run completes. Stare at the event history until replay clicks.

### Day 54 — Agents as distributed systems (the thesis becomes literal)
- **Learn:** map it 1:1 — agent run = workflow; agent step = activity; message history = event log; resume-after-crash = replay (Day 53); tool retries = Day 51; multi-step with side effects = saga (Day 52); long runs off the HTTP path = queue + workers + status (Days 49–50); **agent checkpointing** — serialize agent state after each step so a crash resumes, not restarts (this *is* LangGraph's checkpointer, Day 74 — you're learning the mechanism before the brand); multi-agent communication: direct in-process vs **via queues** (all delivery-guarantee problems included) vs **blackboard** (shared state + concurrency control — Day 39).
- **Case studies:** ① Deep-research-style long-running agents (multi-minute fan-out runs) — necessarily queue-backed, checkpointed, resumable; the product shape *is* this architecture. ② A composite production failure: agent charged a customer (step 4), crashed (step 6), restarted from scratch and charged again — every missing pattern from this phase in one incident. Idempotency + checkpointing or bust.

## Day 58½ — Math you actually need (intuition + code, NO proofs) — do this before Phase 6

You told me you have no mathematical background, and Phase 6 needs a little — but far less than people fear, and none of it as formal proofs. This bridge day builds the *four* ideas the transformer rests on, each as **intuition + a picture + a few lines of NumPy** you can run. If a concept clicks in code, you don't need the proof. Take two or three days on this if you need them.
- **Learn (all visual/code-first):**
  1. **Vectors & the dot product** — a vector is just a list of numbers (a point/arrow in space); the **dot product** measures "how aligned" two vectors are — big when they point the same way, zero when perpendicular. *This is literally how attention scores relevance and how embeddings (Day 44) measure similarity* — you've already used it.
  2. **Matrices & matrix multiplication** — a grid of numbers; multiplying a vector by a matrix **transforms** it (rotate/scale/project). A neural network layer *is* a matrix multiply plus a bias. Get comfortable with shapes (`(m×n)·(n×p)=(m×p)`) — 90% of "dimension mismatch" bugs are shape bugs.
  3. **Derivatives, gradients & the chain rule** — a derivative is just a **slope**: "if I nudge this input, which way and how fast does the output move?" A **gradient** is the slope in every direction at once → it points uphill, so you step *downhill* to reduce error (that's gradient descent, Day 59). The **chain rule** = slopes multiply through a pipeline (that's backpropagation, Day 59). No limits, no proofs — just "slope, and slopes chain."
  4. **Probability & softmax** — a probability distribution is a set of non-negative numbers that sum to 1; **softmax** turns any list of scores into such a distribution (bigger score → bigger share, smoothly). *This is how a model turns raw output numbers into "next-token probabilities" (Day 62) and how attention turns scores into weights (Day 61).* Plus `log`/`exp` intuition (why we work in log-space: turns tiny multiplied probabilities into addable numbers).
- **Build:** a single NumPy notebook, one cell per idea: (a) dot-product of two vectors + verify it's large for aligned / ~0 for perpendicular; (b) multiply a 2D vector by a rotation matrix and plot before/after; (c) take a simple function like `f(x)=x²`, compute its slope numerically (`(f(x+h)-f(x))/h`) and confirm it matches `2x`; (d) implement `softmax([2.0, 1.0, 0.1])` by hand and confirm it sums to 1. If all four run and make sense, you are ready for Phase 6.

# PHASE 6 — LLM Internals: How Models Are Actually Made (Days 59–70)

You've treated the LLM as a black box for 58 days (correctly — that's how you *use* it). Now open it. This is the phase that separates an AI *engineer* from an API *caller*: after it, latency, cost, context limits, hallucination, and fine-tuning stop being mysteries and become mechanisms you can reason about. **Do Day 58½ (the math on-ramp) first.** This phase leans deliberately on **intuition + runnable code over formal derivation** — every idea is taught by building a tiny version and watching it work, not by proving theorems. Heavier than the rest — pace yourself, and build the tiny models; a GPT you trained by hand is worth 1,000 diagrams.

### Day 59 — What "learning" means: from linear regression to neural nets
- **Learn:** the core idea beneath *all* ML — a model is a function with tunable **parameters (weights)**; "training" = adjust weights to minimize a **loss** (how wrong the output is) via **gradient descent** (compute the slope of loss w.r.t. each weight, step downhill); start concrete with linear regression (`y = wx + b` — literally fit a line), then **logistic regression** (add a nonlinearity → classification), then stack them into a **neural network** (layers of weighted sums + nonlinear activations — a universal function approximator); **backpropagation** (the chain rule applied backward through the layers to get every gradient efficiently); epochs, batches, learning rate.
- **Why:** every LLM is this — just very large and with a specific architecture (Day 61). Grasp it on one neuron and the 400-billion-parameter version is only bigger, not different.
- **Case studies:** ① The perceptron → "AI winter" → deep-learning revival (2012 AlexNet) — why *scale + data + GPUs* unlocked what the math had allowed for decades. ② Why GPUs (not CPUs) — training is millions of parallel matrix multiplies (Day 20's parallelism, on silicon built for it).
- **System design (2):** ① Design the training-data pipeline for a supervised model (collection → cleaning → labeling → train/val/test split — and *why leakage between them silently inflates your metrics*). ② Design an experiment-tracking system (runs, hyperparameters, metrics, artifacts — reproducibility as engineering).
- **Build:** implement linear regression **from scratch in NumPy** — forward pass, MSE loss, manual gradients, gradient-descent loop; watch the loss fall and the line fit. Then a 2-layer neural net (with backprop by hand) on a toy classification set. No framework — feel the gradients.

### Day 60 — Language modeling & tokenization: text becomes numbers
- **Learn:** the language-modeling objective — **predict the next token** given all previous (self-supervised: the internet is its own label, no human annotation needed — *this* is why it scaled); why words won't work as units (infinite vocabulary, unseen words) → **subword tokenization** (**BPE**: start from bytes, greedily merge frequent pairs into tokens); the vocabulary; token IDs → **embeddings** (each token → a learned vector — Day 44's embeddings, now as the model's input layer); why tokenization explains real quirks: why models miscount letters ("how many r's in strawberry"), why JSON/code costs more tokens than prose (Day 25), why non-English text costs more, why you're billed per token (Day 23).
- **Case studies:** ① GPT-2/GPT-3's BPE tokenizer and the famous glitch tokens (`SolidGoldMagikarp`) — rarely-trained tokens causing bizarre behavior; tokenization as an attack/failure surface. ② Multilingual token inequity — the same sentence costing 2–3× more tokens in Hindi/Thai than English, and the fairness/cost implications.
- **System design (2):** ① Design a cost estimator for an LLM feature (tokens-in × input-price + tokens-out × output-price across a realistic request mix — the model behind Day 34's spend caps). ② Design chunking (Day 45) that respects *token* boundaries, not character counts — and why that matters for context packing.
- **Build:** implement **BPE from scratch** — train it on a text corpus (merge frequent pairs), encode/decode, inspect the learned vocab; compare your token counts to `client.messages.count_tokens` on the same strings and explain the differences.

### Day 61 — The Transformer I: attention (the idea that changed everything)
- **Learn:** the problem with older sequence models (RNNs process tokens one-by-one → slow, and they *forget* long-range context); **self-attention** — every token looks at every other token and pulls in what's relevant (the mechanism that made long context *and* parallel training possible); **Q/K/V** intuition (Query: what am I looking for; Key: what do I offer; Value: what I'll actually pass along — attention weight = softmax of Q·K, output = weighted sum of V); **multi-head attention** (several attention patterns in parallel — syntax, coreference, etc.); why it's parallelizable (all positions at once — Day 20/GPU payoff) and why it's **O(n²)** in sequence length (the root cause of context limits and why long contexts cost quadratically — Day 43's "context rot" economics).
- **Case studies:** ① "Attention Is All You Need" (2017) — read the actual paper now that you have the vocabulary; it's shorter and more readable than its reputation. ② The context-length arms race (2k → 200k → 1M tokens) and the tricks that fight O(n²) (FlashAttention's IO-aware kernels, sparse/sliding attention) — verify current state.
- **System design (2):** ① Reason about the latency/cost/memory of a 100k-token context request vs 5k (quadratic attention + linear KV cache — Day 63) — build the mental cost model. ② Design a context-packing strategy under an O(n²) reality (what earns a token's place in the window — recency, relevance/RAG, summary — Day 43).
- **Build:** implement **single-head self-attention from scratch in NumPy** on a toy sequence — build Q/K/V, compute the attention matrix, softmax it, produce outputs; *visualize the attention weights as a heatmap* and see which tokens attend to which. This is the load-bearing hour of the phase.

### Day 62 — The Transformer II: the full architecture
- **Learn:** assemble the block — **positional encoding** (attention is order-blind; you must inject position — learned vs RoPE); the **transformer block** (multi-head attention → add & **layer-norm** → **feed-forward** network → add & norm; residual connections that let gradients flow through deep stacks); **causal masking** (a decoder can't peek at future tokens — that's what makes it a next-token predictor); stacking N blocks; the **output head** (final layer → logits over the vocabulary → softmax → probabilities); **decoding/sampling** — greedy vs temperature vs top-k/top-p (and the honest note from your stack: current Claude models *remove* `temperature`/`top_p` — you steer via prompting/effort — Day 25); "context rot" now explained mechanically (attention diluted across more tokens; positional effects).
- **Case studies:** ① The GPT decoder-only lineage vs BERT (encoder-only) vs T5 (encoder-decoder) — why "decoder-only + scale" won for generative use. ② Sampling-parameter incidents — a temperature-too-high support bot going off the rails vs temperature-0 giving robotic identical answers; why the parameter is a product decision.
- **System design (2):** ① Design an inference request's full path (tokenize → prefill the prompt → decode tokens autoregressively → detokenize → stream — Day 33's SSE); mark where latency accrues. ② Design output validation/guardrails on a probabilistic generator (schema checks, Day 25's constrained decoding, refusal/repair loops).
- **Build:** **nanoGPT** — train a tiny character-level GPT (Karpathy's, or assemble from your Day 61 attention) on a small text file (e.g. your own notes); watch it go from noise → words → sentences as loss drops; sample from it at different "temperatures" (in your from-scratch decoder, where the param still exists) and *see* the effect.

### Day 63 — Inference internals: KV cache, batching, quantization
- **Learn:** why the first token is slow but the rest stream (the **prefill** vs **decode** split — prefill processes the whole prompt in parallel; decode is one-token-at-a-time and *memory-bandwidth-bound*); the **KV cache** — cache past keys/values so each new token doesn't recompute the whole sequence; its cost = the real memory eater (linear in context × layers × batch — Day 4's memory budgeting, now for GPUs), and *why long contexts and big batches fight for VRAM*; **continuous batching** (vLLM/TGI: pack many requests through the GPU efficiently — the throughput unlock); **quantization** (fp16 → int8/int4: smaller/faster weights, small quality cost); **speculative decoding** (a small model drafts, the big one verifies — free-ish speedup); this is *exactly* why **prompt caching** works (Day 64) — a reusable KV prefix.
- **Case studies:** ① vLLM's **PagedAttention** — applying OS virtual-memory paging (Day 4!) to the KV cache to stop VRAM fragmentation; the paper is a beautiful backend-meets-ML crossover. ② Why hosted inference is priced as it is (a GPU-hour serving N concurrent requests via batching — the unit economics behind every per-token price).
- **System design (2):** ① Design a self-hosted inference service (GPU memory budget: weights + KV per concurrent request; max batch size; queueing when full — Day 33/49). ② Decide build-vs-buy: self-host an open model vs a hosted API for three scenarios (privacy-critical, spiky low-volume, steady high-volume) — TCO math.
- **Build:** measure it on a real hosted model — time-to-first-token vs inter-token latency across prompt sizes (prefill cost is visible); compute the KV-cache memory a 100k-context request would need for an open 7B model (params × layers × bytes) and write down why batching + paging exist.

### Day 64 — Prompt caching, context engineering & cost control at the token level
- **Learn:** **prompt caching** as the KV-cache (Day 63) exposed as an API feature — a stable prompt *prefix* (system prompt, tool defs, RAG context, few-shot examples) is cached provider-side; identical prefixes on later calls skip prefill → big cost + latency savings; the engineering consequence: **order your prompt stable-prefix-first, volatile-suffix-last** (put the changing user turn at the *end*), and structure agent loops to preserve the cacheable prefix; **context engineering** as the discipline — every token earns its place (Day 43); **model routing/cascades** (cheap model triages → escalate hard cases to the expensive one — Day 35's economics); **semantic caching** (Day 45) revisited as cost control. Consult the `claude-api` skill for the current caching interface before building.
- **Case studies:** ① Published prompt-caching results (multi-× cost cuts and latency drops on cache hits) — verify current pricing/mechanics against the docs. ② A model-cascade in production (e.g. cheap-classifier-then-expensive-reasoner) cutting bills by an order of magnitude while holding quality — the pattern behind most "we reduced LLM cost 90%" posts.
- **System design (2):** ① Design the prompt layout of your platform agent for maximum cache hits (what's in the stable prefix, what must stay volatile, how tools/RAG context fit) — and measure the hit rate. ② Design a model-routing layer (features → cheap/expensive decision, confidence-based escalation, per-tier cost budget, quality monitoring).
- **Build:** restructure your platform agent's prompt for prefix caching; measure cost/latency per turn before vs after (real numbers). Add a cheap-model triage step in front of the expensive agent; on your Day 45 golden set, report cost/quality with and without the cascade.

### Day 65 — How frontier models are trained: pretraining → alignment
- **Learn:** the pipeline that turns Day 62's architecture into ChatGPT/Claude — **pretraining** (next-token prediction over trillions of tokens; data curation, dedup, filtering; the scale/compute/data story — "scaling laws"); **supervised fine-tuning (SFT)** (teach it to *follow instructions* on curated demonstrations — turns a document-completer into an assistant); **preference alignment** — **RLHF** (humans rank outputs → train a reward model → optimize the policy toward it) and **DPO** (skip the separate reward model — simpler, now common); what alignment does and doesn't fix; **why models hallucinate** (trained to produce *plausible* continuations, not to *know*; no built-in truth signal — the mechanism behind why RAG/citations/guardrails, Days 45/89, are architectural necessities not band-aids).
- **Case studies:** ① InstructGPT (the RLHF paper) — the same base model going from unusable to helpful via alignment; the result that made the assistant paradigm. ② Constitutional AI (Anthropic) — using a written constitution + AI feedback to reduce human labeling of harmful content; alignment method as a design choice with product consequences.
- **System design (2):** ① Design a preference-data collection system (pairwise comparisons UI, annotator agreement, quality control, bias auditing — the human infrastructure behind alignment). ② Design "hallucination defense in depth" for a factual product (retrieval grounding + citations + a verifier pass + confidence-gated "I don't know" + human escalation).
- **Build:** you won't run RLHF, but *simulate the shape*: take your nanoGPT (Day 62), do a tiny "SFT" by fine-tuning it on a small instruction-formatted dataset, and observe behavior change; then hand-rank a few outputs and write the reward-model + DPO objective in pseudocode-that-runs on toy data, so the mechanism is concrete, not mythical.

### Day 66 — Fine-tuning in practice: LoRA/QLoRA & when NOT to
- **Learn:** the decision framework *first* (the senior skill) — **prompting → RAG → fine-tuning**, in that order of reach-for; fine-tuning is right for *form/behavior* (rigid output format, tone, a narrow high-volume task moved to a smaller/cheaper model, latency reduction) and *wrong* for *adding knowledge* (that's RAG — Day 45); **full fine-tuning** (update all weights — expensive, needs the whole model in memory) vs **PEFT/LoRA** (freeze the base, train tiny low-rank adapter matrices — ~0.1% of params, cheap, swappable per task) vs **QLoRA** (quantize the base to 4-bit + LoRA — fine-tune a big model on one consumer GPU); **distillation** (big model's outputs train a small cheap one); data quality ≫ quantity; catastrophic forgetting; evaluation *before and after* (Day 80).
- **Case studies:** ① The LoRA paper + the open ecosystem of swappable adapters (one base, many task adapters served together) — why PEFT democratized fine-tuning. ② A "we fine-tuned to add company knowledge and it hallucinated worse" cautionary tale → the fix was RAG; fine-tuning taught *format*, RAG supplied *facts*. Internalize the division of labor.
- **System design (2):** ① Design a fine-tuning pipeline (data curation → train → eval-gate → versioned adapter registry → canary deploy — Days 80/88). ② Design a multi-tenant serving stack with per-tenant LoRA adapters over one shared base (memory savings, adapter routing, isolation).
- **Build:** QLoRA fine-tune a small open model (e.g. a 1–3B model via `peft`+`transformers` on Colab/free GPU, or a genuinely tiny local one) to enforce a strict output format your prompting struggled with; **evaluate before/after** on a held-out set (format-adherence rate); write the one-paragraph verdict: was fine-tuning the right tool here vs better prompting?

- **Learn:** how models see/hear — **vision transformers** (an image → patches → tokens → the same transformer; multimodal models fuse text + image tokens in one sequence); the practical surface of multimodal (document understanding, screenshots, charts — Day 90 domain relevance); **embeddings revisited as their own model class** (contrastive training pulls similar pairs together / pushes dissimilar apart — the training behind Day 44's vectors; matryoshka/variable-dim embeddings; domain-specific vs general); the **model-selection framework** — proprietary (Claude/GPT/Gemini) vs open-weight (Llama/Qwen/Mistral — verify current), and the axes: capability, context, latency, cost, privacy/on-prem, licensing, tool-use quality; benchmarks and **why leaderboards mislead** (contamination, gaming, benchmark≠your task — you evaluate on *your* data, Day 80).

### Day 79 — Agent security: prompt injection, tool poisoning & the lethal trifecta
- **Learn:** **prompt injection** as the defining unsolved threat — untrusted content (web pages, docs, emails, tool outputs, MCP resources — Day 78) carrying instructions the model may obey; direct vs **indirect** injection (the dangerous one: poison a document the agent will later read); the **"lethal trifecta"** (private-data access + untrusted-content exposure + external communication = exfiltration risk — avoid combining all three unsupervised); **tool poisoning** (a malicious MCP server / compromised tool returning crafted data); the **confused-deputy** problem (the agent has more authority than the attacker, who tricks it into using that authority); why it's structurally hard (Day 62 — no reliable trusted/untrusted channel separation in one token stream); **defense in depth**: treat *all* tool/RAG/MCP output as untrusted data (Day 30), enforce **least-privilege with real backend RBAC/authz** (Day 32 — not prompt pleading), guardrails as a separate step (Day 69), human gates on irreversible actions (Day 55), sandboxing (Day 8), and — critically — **OWASP-style API security** underneath it all (**BOLA**: check object ownership on every tool-backed access; injection classes; secrets in Key Vault + managed identity).
- **Case studies:** ① Real indirect-injection demos (hidden instructions in a webpage/email/PR making an agent exfiltrate data or misuse a tool — verify current examples) and the **OWASP Top 10 for LLM Applications** (read it as the working checklist). ② The GM $1-truck / Air Canada refund incidents (Days 55/69) reframed as *security + authorization* failures — the model's output is an authenticated action unless the backend says otherwise.
- **System design (2):** ① Threat-model your platform agent end to end (trust boundaries: user, model, tools, RAG, MCP; where untrusted data enters; where authority lives; where the trifecta could combine) and design the mitigations per boundary. ② Design least-privilege tool authorization (per-tool, per-user scopes enforced in the backend — Day 32; BOLA checks on every object access; secrets never in the prompt).
- **Build:** **red-team your own platform** (Days 74/78 agent): (a) indirect prompt injection via a tool's returned data attempting to make the agent take an unauthorized action; (b) a BOLA attack where the agent is steered to access another user's resource through a tool. Then add the defenses (backend RBAC + object-ownership checks + output guardrail + trifecta-breaking constraint) and prove both attacks now fail. Keep the exploit + fix write-up — it's portfolio gold.

# PHASE 8 — Production Engineering & Operations (Days 85–93)

### Day 86 — Databases in production: migrations, scaling, operations

### Day 87 — Testing strategy: from units to load
- **Learn:** the **test pyramid** (many fast unit tests → fewer integration → few e2e — and the "ice-cream cone" anti-pattern that inverts it); **pytest** fluency (fixtures, parametrization, `dependency_overrides` to mock DB/model/tools — Day 28's DI payoff); **integration testing with testcontainers** (real Postgres/Redis in a container, not mocks — Day 2, catches what mocks hide); **testing async** (`pytest-asyncio`); **testing non-deterministic LLM code** (you don't assert exact strings — you assert *properties*, use recorded fixtures/VCR for tool calls, and lean on the eval suite — Day 80 — as the "test" for agent quality); **load testing** (Locust/k6 to find the real breaking point — Days 16/21) and reading the results (where does p99 explode, what saturates first — Day 85).

- **Build:** build the test suite for your platform — unit tests with `dependency_overrides` mocking the model/tools, integration tests against real Postgres+Redis via testcontainers, property-based assertions for the agent's structured outputs, and a Locust load test that finds *where your platform breaks first*. ### Day 88 — Containers, CI/CD & eval-gated deployment
- **Learn:** **Docker** properly (Day 2 payoff — a container is a namespaced/cgrouped *process*, not a VM): layered image filesystems, **build-cache ordering** (rarely-changing deps before frequently-changing code — else every build is slow), multi-stage builds (small final images), non-root, `.dockerignore`, image scanning (Day 89); **CI/CD** as a non-negotiable automated gate — the pipeline: lint (ruff) → type-check (mypy) → unit tests → integration tests (testcontainers) → build image → **agent eval suite (Day 80) → red-team tests (Day 79)** → deploy; **deployment strategies** — blue-green, **canary** (small % first — Day 16's draining), rolling, feature flags; **rollback** as a first-class capability (fast, tested, one command); for agents specifically: eval-gate + prompt-versioning (Day 68) so a quality regression blocks the deploy exactly like a failing unit test.
- **Case studies:** ① A prompt/model change that passed all code tests but tanked answer quality → why the **eval gate** exists (Day 80); the deploy it *should* have blocked. ② A canary that caught a regression at 5% traffic and auto-rolled-back — canary + good metrics (Day 85) as the safety net.
- **System design (2):** ① Design the full CI/CD pipeline for your agent platform (every gate above, what blocks deploy, canary + auto-rollback triggers on SLO burn — Day 92). ② Design a deployment strategy for a risky prompt/model change (shadow/canary, eval gate, A/B with online eval — Day 80, instant rollback).
- **Build:** containerize your platform (multi-stage, non-root, cache-ordered Dockerfile) and build a **complete CI/CD pipeline** (GitHub Actions): lint → type → unit → integration → build → **eval suite (Day 80) → red-team (Day 79)** → deploy, where a regression at *any* stage blocks the deploy. Prove it: push a prompt change that fails the eval gate and watch the pipeline stop the deploy.

### Day 89 — Security hardening: the full stack
- **Learn:** consolidate every security thread into a working checklist — **OWASP API Security Top 10** (esp. **BOLA** — object-ownership checks on every access, Day 32; broken auth; excessive data exposure; injection — parameterized queries *always*, Day 38) **+ OWASP Top 10 for LLM Apps** (prompt injection, insecure output handling, excessive agency, sensitive-info disclosure — Days 69/79); **secrets management** (Key Vault / managed identity — *never* secrets in code, env files in git, or prompts; rotation); **the principle of least privilege everywhere** (DB users, service identities, tool scopes — Day 79); **dependency/supply-chain security** (Day 7 — lockfiles, SCA scanning, SBOM, image scanning — Day 88); **network security** (Days 9–16 payoff — private subnets, security groups, TLS everywhere, mTLS internal — Day 14); **defense in depth** as the organizing principle (no single control is trusted alone).
- **Case studies:** ① A real BOLA breach (verify current — the recurring "change the ID in the URL, see someone else's data" class) — the single most common serious API vuln, and a one-line-check fix. ② A supply-chain compromise (xz/event-stream-class — Day 7) → why lockfiles + SCA + minimal images + least privilege compound into resilience.
- **System design (2):** ① Full threat model of your platform (STRIDE-style across every trust boundary — user, API, agent, tools, RAG, MCP, DB, model provider; Days 32/79 fused) with the control per threat. ② Design secrets + identity architecture (managed identities, Key Vault, per-service least-privilege roles, rotation, zero long-lived credentials).
- **Build:** security-harden your platform and *prove* it: fix any BOLA gaps (object-ownership checks on every resource + tool access — write the failing-then-passing tests, Day 32), move all secrets to a vault/managed identity (none in code or env-in-git), add dependency + image scanning to CI (Day 88), enforce TLS. Re-run your Day 79 red-team + add API-level attacks (BOLA, injection, auth bypass) — everything must fail safely.

### Day 90 — Cloud, infrastructure & deployment platforms
- **Learn:** the cloud primitives under everything (compute / storage / network / identity — you've used them; now name them); **choosing compute** — serverless functions (event-driven, scale-to-zero, **cold starts** — why they happen: a new worker + slow first init + a slow first model call compound for agents; mitigations — Day 63) vs **containers/managed container platforms** (always-on, scale-to-zero-ish — Container Apps/Cloud Run/ECS) vs Kubernetes (only when you truly need its control — most don't) vs managed PaaS; **IaC** (Terraform/Bicep — infrastructure as versioned, reviewed, reproducible code, never click-ops); **autoscaling** (on what signal — CPU? queue depth? — Day 49 — request concurrency?); **cost engineering** (the cloud bill as an SLO: right-sizing, scale-to-zero, spot, the "our LLM+GPU bill is the real cost" reality — Days 63/81); multi-region basics (Days 10/12/47) at an awareness tier.
- **Case studies:** ① A serverless cold-start-latency story for an LLM app (cold worker + model init + first token → seconds) and the fix (min-instances/pre-warming, smaller images — Day 88, lazy imports). ② A cloud-cost blowup (idle over-provisioned GPUs / no scale-to-zero / runaway agent loops — Day 24) → the FinOps discipline that fixed it.
- **System design (2):** ① Choose the compute platform for each component of your platform (API, agent workers — Day 54, background jobs — Day 49, inference) and defend each; design the autoscaling signal for each. ② Design multi-region for the platform (what's active-active vs active-passive, data replication — Day 47, failover via DNS — Day 12, and the honest RTO/RPO).
- **Build:** deploy your platform to a real cloud via **IaC** (Terraform/Bicep — the infra is code, reviewed and reproducible): API + agent workers + Postgres + Redis + queue, with autoscaling on the right signal (queue depth for workers — Day 49) and scale-to-zero where sensible. Load-test the deployed system; measure cold-start impact on agent latency and mitigate it.

### Day 91 — Data & ML lifecycle in production (MLOps/LLMOps)
- **Learn:** the operational loop around the model layer (Phase 6 in production) — **the feedback flywheel** (production traces — Day 82 → flagged failures → curated datasets → eval — Day 80 → prompt/model/RAG improvements → deploy — Day 88 → repeat); **data pipelines** (batch vs streaming — Day 49; the RAG ingest/re-index pipeline as a first-class, monitored system — Days 45/52); **prompt & model versioning** (Day 68 — everything versioned, nothing edited-in-place); **A/B testing & online eval** in production (Day 80 — measuring real quality, not vibes); **drift detection** (input distribution shifts, quality decay over time — the model didn't change but the world did); **the RAG index as a living system** (freshness SLA, incremental updates via outbox/CDC — Day 52, staleness monitoring); dataset & experiment lineage (Day 59).
- **Case studies:** ① A team's LLMOps flywheel (verify current) — production failures continuously becoming eval cases becoming improvements; the compounding moat. ② A RAG-staleness incident (KB updated, index didn't, agent confidently cited old policy — Days 45/52) → the CDC-driven freshness pipeline that fixed it.
- **System design (2):** ① Design the full feedback flywheel for your platform (trace → flag → dataset → eval → improve → deploy) as running infrastructure, not a manual process. ② Design RAG freshness end-to-end (source change → outbox/CDC — Day 52 → re-chunk/re-embed → cache invalidation — Day 42 → agent sees new fact) with a measurable SLA and monitoring.
- **Build:** close the loop on your platform — wire production agent traces (Day 82) so low-quality/expensive/flagged trajectories auto-flow into a review queue → curated into your Day 80 golden set → re-run eval in CI (Day 88). Then automate RAG re-indexing on source change (outbox → worker → re-embed → invalidate cache) and monitor index freshness. Demonstrate one full turn of the flywheel.

### Day 92 — SRE: SLIs, SLOs, error budgets & incident response
- **Learn:** the SRE discipline that ties it together — **SLIs** (indicators that reflect *real user experience* — latency, availability, correctness — not vanity metrics), **SLOs** (the target, e.g. 99.9% of requests <500 ms), **error budgets** (100% − SLO = your license to take risks; spend it shipping, freeze features when it's exhausted — the elegant reframing of the dev-vs-ops tension); **agent-specific SLIs** (task-success rate — Day 80, cost-per-task — Day 81, escalation rate, p99 latency — using trajectory data, Day 82); **incident response** (severity levels, incident commander, comms, the timeline); **blameless postmortems** (every case study in this course *is* one — now you write them: root cause, contributing factors, action items, no blame); **toil reduction** (automate the repetitive — Day 88).
- **Case studies:** ① Google's SRE book chapters on SLOs + error budgets (the source; read the error-budget chapter). ② Any blameless postmortem you've read this course (GitHub 2018 — Day 47, Slack 2021 — Day 16, Meta 2021 — Day 10) reread *as a template* for the postmortems you'll now write.
- **System design (2):** ① Define real SLIs/SLOs/error budgets for your platform (both backend SLIs and agent SLIs — Days 80/81; how each is measured from your Day 85 telemetry; what the error budget gates). ② Design incident response for an agent-specific incident (agent doing something harmful/expensive — Days 79/81; detection, kill-switch, comms, postmortem).
- **Build:** define and instrument SLIs/SLOs for your platform (availability, p99 latency, agent task-success, cost-per-task) with error-budget tracking on your Day 85 dashboard + SLO-burn alerts; then run a **game-day** — inject a failure (kill a dependency — Day 51, exhaust a pool — Day 40, break the model API), respond as incident commander, and *write the blameless postmortem*. That postmortem is a portfolio artifact.

# PHASE 9 — Domain Systems & Capstone (Days 94–100)

You now have the full stack — machine → network → concurrency → API → data → distributed → model → agent → production. This phase applies it to the three hard domains you named (financial, real-time, prediction), then converges everything into one shipped, production-grade agentic platform.

### Day 94 — Financial & transactional systems
- **Learn:** why money is the strictest domain — **correctness is non-negotiable** (a lost update — Day 39 — is a lost dollar); the **double-entry ledger** (append-only — Day 52's event sourcing! — every transaction is two balanced entries; balance = fold of the log; you *never* mutate a balance in place); **money representation** (integer minor units or Decimal — *never* floats: `0.1 + 0.2 ≠ 0.3` will cost you real money); **idempotency as law** (Days 31/50 — every payment operation is idempotent or it's a bug); **exactly-once effect** (Day 50) for charges; **reconciliation** (comparing your ledger to the payment provider's — trust but verify); **consistency over availability** (Day 41 — CAP: a bank chooses C; refuse the transaction rather than double-spend); audit trails & compliance (immutable logs — Day 82; regulatory retention); **saga + compensation** (Day 52) for multi-step financial flows (charge → fulfill → refund-on-failure).
- **Case studies:** ① How Stripe/ledger systems model money (double-entry, idempotency, immutable events — verify current writeups) — the reference architecture for correctness. ② A floating-point money bug (rounding errors accumulating into real losses) → the integer/Decimal fix; and a double-charge incident → the idempotency fix (Days 31/50).
- **System design (2):** ① Design a payment ledger (double-entry, append-only events — Day 52, idempotent operations — Day 31, balance projections — CQRS Day 52, reconciliation job) that *cannot* lose or duplicate money. ② Design a fraud-detection pipeline (real-time scoring — Day 95, rules + ML — Day 96, human review queue — Day 55, feedback loop — Day 91) — bridging into the next two days.
- **Build:** build a correct **double-entry ledger service** (append-only transaction events → balance projection; integer minor units; idempotency keys on every operation; a reconciliation check) with tests that hammer it with concurrent + duplicate + retried operations (Day 39's anomaly tests + Day 50's duplicate tests) and *prove* no money is lost or created.

### Day 95 — Real-time systems
- **Learn:** what "real-time" means here (low-latency, event-driven, often stateful-connection systems — the place the stateless thesis *bends*, Day 57); **WebSockets** deep (Day 33 — the bidirectional upgrade; connection = state; **consistent-hash routing** of connections to nodes — Day 48; what happens on node death; heartbeats/reconnection); **presence & pub/sub fan-out** (Day 49 — broadcasting to millions of connections; the fan-out-on-write vs fan-out-on-read trade — Day 57); **backpressure** (Day 21 — a slow consumer can't be allowed to sink the system); **event streaming** (Kafka — Day 49 — for real-time pipelines; windowing, stream processing basics); **latency budgets** (Day 1 — every millisecond allocated); real-time + agents (streaming agent responses — Day 33, voice — Day 83, live tool feedback).
- **Case studies:** ① Discord/Slack real-time architecture (Day 57 payoff — WebSocket gateways, consistent-hash routing, presence at scale). ② A trading/live-scoreboard/collaborative-editing system's latency + consistency trade-offs (where they sacrifice what — Day 41).
- **System design (2):** ① Design a real-time collaboration backend (WebSocket gateways + consistent-hash routing — Day 48 + conflict resolution — CRDT/OT at awareness tier + presence + reconnection). ② Design a real-time market-data/leaderboard fan-out (millions of subscribers, sub-100ms updates, backpressure, the fan-out trade-off).
- **Build:** build a real-time system — a WebSocket-based live feature (collaborative counter/chat/live-dashboard) with multiple gateway instances, consistent-hash connection routing (Day 48), presence, heartbeats, and graceful reconnection; load-test with many concurrent connections (Day 87) and demonstrate a gateway node dying with clients cleanly reconnecting elsewhere.

### Day 96 — Prediction & intelligent systems
- **Learn:** prediction systems *beyond* LLMs (the ML you'll wrap in backends) — the **recommendation/ranking** shape (candidate generation → ranking → re-ranking — structurally *identical* to your RAG pipeline, Day 45!); **feature engineering & the feature store** (offline training features must match online serving features — the **train/serve skew** bug); **real-time inference** (Day 63 — a model behind a low-latency API; batching; caching predictions — Day 42); **time-series forecasting** basics (at awareness tier); **the offline→online gap** (a model great in the notebook, useless in prod — data drift, Day 91, skew, latency); **classical ML vs LLM** (when a cheap gradient-boosted model beats an LLM — structured tabular prediction, latency/cost-critical paths — the senior judgment); **hybrid intelligent systems** (rules + ML + LLM layered — e.g. fraud, Day 94: fast rules filter → ML scores → LLM explains/handles edge cases).
- **Case studies:** ① A recommendation system architecture (Netflix/YouTube-class — candidate gen → ranking; the RAG-parallel is exact — Day 45). ② A train/serve-skew incident (offline AUC great, online performance terrible → features computed differently in the two paths) → the feature-store fix.
- **System design (2):** ① Design a recommendation/ranking system (candidate generation → feature retrieval → ranking model → re-rank → serve with caching — Days 42/45; feedback loop — Day 91). ② Design a hybrid fraud/decision system (rules → ML score → LLM-for-edge-cases + explanation, with human review — Day 55, and a feedback flywheel — Day 91).
- **Build:** build a prediction service — train a simple classical model (gradient boosting on a real tabular dataset), serve it behind a low-latency FastAPI endpoint with prediction caching (Day 42) and monitoring (Day 85), then build a **hybrid** endpoint (fast rules → ML score → escalate ambiguous cases to your LLM agent for handling + explanation). Measure latency and the rules/ML/LLM split of traffic.

### Day 97 — Capstone I: architecture, blueprint & foundation
- **The capstone (Days 97–100):** ship **one complete, production-grade agentic platform** in a domain you care about (e.g. an agentic financial-operations assistant, a real-time support platform, a research-and-decision system) that integrates the *entire* course. ### Day 98 — Capstone II: the agentic core

### Day 99 — Capstone III: hardening, evaluation & production

# Final word

This is a **100-day sequence, not a 100-day deadline.** If a day needs two, take two — the ordering is the value, not the calendar. By Day 100 you won't have "studied backend and agentic AI."
